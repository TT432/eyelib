# Feature Landscape — Gradle Task Automation for ClientSmoke

**Domain:** One-click smoke test launch via Gradle
**Researched:** 2026-05-08

## Table Stakes

Features users expect. Missing = product feels incomplete.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Single gradle command to run all smoke tests | Core v1.1 requirement; current 3-step process is friction | Low | MDGL auto-generates `runClientSmoke` from run name |
| Auto-enable smoke test configuration | Manual `enabled=true` in TOML is error-prone | Low | `taskBefore` writes config file before launch |
| Isolated game directory | Don't pollute normal dev world with smoke test artifacts | Low | `gameDirectory = project.file('run-smoke')` |
| Auto-exit after tests complete | Smoke test must terminate, not leave Minecraft open | Low | Already built into state machine; `jvmArgument` as belt-and-suspenders |
| No impact on normal `runClient` | Smoke test must be opt-in, not break daily dev workflow | Low | Separate run config; `enableSmokeTest` defaults to `false` |

## Differentiators

Features that set product apart. Not expected, but valued.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| IDE run configuration auto-generated | One click in IntelliJ to run smoke tests | Low | `ideName`/`ideFolderName` in run config |
| Config file regeneration on each run | Never stale; TOML always matches expectation | Low | `taskBefore` runs every launch |
| Isolated config scope | Smoke test config lives in `run-smoke/config/`, never touches prod `run/config/` | Low | Automatic via `gameDirectory` |
| Elevated logging during smoke run | `logLevel = DEBUG/INFO` for easier CI debugging | Low | `logLevel = org.slf4j.event.Level.INFO` |

## Anti-Features

Features to explicitly NOT build.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| Custom Gradle plugin (buildSrc or standalone) | Overkill for one run config; adds maintenance burden | MDGL `runs` DSL + `tasks.register()` |
| `Exec`/`JavaExec` task subclass wrapping MDGL | MDGL handles Forge classpath remapping, NFRT, userdev runtime — duplicating is fragile | Let MDGL generate `runClientSmoke` |
| `gradle.properties` default change to `enableSmokeTest=true` | Would make all `runClient` invocations load smoke test mod (unwanted overhead) | Keep default `false` |
| Runtime dependency on clientsmoke (not `localRuntime`) | `runtimeOnly` is transitive — downstream consumers would inherit the smoke test mod | Keep `localRuntime` (opt-in, non-transitive) |
| System property override for Forge config | `ForgeConfigSpec` reads from TOML file only — system properties are ignored | Write TOML file via pre-task |
| Auto-enable via `gradle.startParameter.projectProperties` mutation | Mutates shared build state, breaks config cache, affects other tasks | Explicit `-PenableSmokeTest=true` CLI flag |

## Feature Dependencies

```
runClientSmoke task (auto-generated by MDGL)
    ├── prepareSmokeConfig pre-task (writes TOML)
    │       └── run-smoke/config/clientsmoke-common.toml (output file)
    ├── enableSmokeTest=true gradle property
    │       └── localRuntime project(':eyelib-clientsmoke') (dependency)
    └── clientSmoke run definition (legacyForge.runs block)
            ├── gameDirectory = run-smoke/
            ├── systemProperty forge.enabledGameTestNamespaces
            ├── jvmArgument -Dclientsmoke.autoExit=true
            └── loadedMods (default: all mods)
```

## MVP Recommendation

Prioritize:
1. `clientSmoke` run definition in root `legacyForge.runs` (core deliverable)
2. `prepareSmokeConfig` pre-task (eliminates manual TOML editing)
3. Documentation: `gradlew runClientSmoke -PenableSmokeTest=true` (user-facing)

Defer:
- Auto-enable of `enableSmokeTest` property: Requires settings-level changes; `-P` flag is sufficient for v1.1
- CI timeout/cleanup handling: Separate v1.2+ concern
- Multi-mod smoke testing (consumer mods): Out of scope for v1.1
