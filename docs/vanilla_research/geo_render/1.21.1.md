# Minecraft 1.21.1 几何渲染管线

## 概述

本文档分析 Minecraft 1.21.1（NeoForge 映射表，源码在 `net/minecraft/` 与 `com/mojang/blaze3d/` 下）原版如何把"方块/实体的几何定义"一路转换为屏幕上的像素。整个管线分为两条相对独立的链路：

- **方块几何管线**：JSON 模型 → `BlockModel` 反序列化 → `FaceBakery` 烘焙成 `BakedQuad` → `SimpleBakedModel` → `SectionCompiler` 把整个 chunk section 网格化为 `MeshData` → 上传到 `VertexBuffer` → `LevelRenderer.renderSectionLayer` 按 `RenderType` 逐层绘制。
- **实体几何管线**：`LayerDefinitions` 编程式注册 `LayerDefinition` → `EntityModelSet.bakeLayer` 烘焙出 `ModelPart` 树 → `LivingEntityRenderer.render` 在每帧驱动 `ModelPart.render` 把立方体写入 `VertexConsumer` → `RenderLayer` 叠加 → `MultiBufferSource.BufferSource` 批量提交。

两条管线共享同一套底层基础设施：`RenderType`/`RenderStateShard`（渲染状态描述）、Blaze3D 顶点缓冲系统（`VertexFormat`/`BufferBuilder`/`MeshData`/`VertexBuffer`）、`ShaderInstance`（着色器与 Uniform）、以及 `LevelRenderer`/`GameRenderer` 的主渲染循环。

> 本文基于 `.local_ref/mc/1.21.1/sources/` 实际源码撰写，类名/方法名/行号均来自该源码。NeoForge 在原版类中插入了若干 `net.neoforged.*` 扩展点，但渲染主干仍是原版逻辑。

---

## 1. 方块几何管线

### 1.1 模型定义层

方块模型以 JSON 文件形式存在于资源包中（`assets/<ns>/models/block/*.json`），由 `BlockModel` 类反序列化。`BlockModel` 实现了 `UnbakedModel` 接口（`net.minecraft.client.resources.model.UnbakedModel`）。

#### 1.1.1 反序列化入口

`BlockModel` 内部持有一个静态 `Gson` 实例（`BlockModel.java:54`），注册了一组 `JsonDeserializer`：

```java
static final Gson GSON = new GsonBuilder()
    .registerTypeAdapter(BlockModel.class, new BlockModel.Deserializer())
    .registerTypeAdapter(BlockElement.class, new BlockElement.Deserializer())
    .registerTypeAdapter(BlockElementFace.class, new BlockElementFace.Deserializer())
    .registerTypeAdapter(BlockFaceUV.class, new BlockFaceUV.Deserializer())
    .registerTypeAdapter(ItemTransform.class, new ItemTransform.Deserializer())
    .registerTypeAdapter(ItemTransforms.class, new ItemTransforms.Deserializer())
    .registerTypeAdapter(ItemOverride.class, new ItemOverride.Deserializer())
    .create();
```

加载入口是 `BlockModel.fromStream(Reader)`（`BlockModel.java:82`），它委托给 `ExtendedBlockModelDeserializer.INSTANCE`（NeoForge 扩展的反序列化器，支持 `neoforge_data` 字段，但最终仍产出 `BlockModel`）。

`BlockModel.Deserializer.deserialize`（`BlockModel.java:336`）依次解析：
- `parent`：父模型 ResourceLocation（`getParentName`）
- `elements`：`List<BlockElement>`（`getElements`）
- `textures`：`Map<String, Either<Material, String>>`（`getTextureMap`）—— 值要么是 `Material`（实际纹理），要么是 `#引用` 字符串
- `ambientocclusion`：`Boolean`（可选）
- `display`：`ItemTransforms`（各 `ItemDisplayContext` 下的变换）
- `gui_light`：`GuiLight`（`FRONT`/`SIDE`）
- `overrides`：`List<ItemOverride>`

#### 1.1.2 BlockModel 结构

`BlockModel`（`BlockModel.java:50`）的核心字段：

| 字段 | 类型 | 含义 |
|------|------|------|
| `elements` | `List<BlockElement>` | 几何元素列表（长方体） |
| `textureMap` | `Map<String, Either<Material, String>>` | 纹理变量表（`#xxx` 引用链） |
| `parent` | `BlockModel` | 父模型（用于继承 elements/textures/transforms） |
| `parentLocation` | `ResourceLocation` | 父模型位置 |
| `transforms` | `ItemTransforms` | 物品展示变换 |
| `overrides` | `List<ItemOverride>` | 物品 override（视状态切换模型） |
| `guiLight` | `GuiLight` | GUI 光照方向 |
| `hasAmbientOcclusion` | `Boolean` | 是否启用环境光遮蔽 |

模型通过 `resolveParents`（`BlockModel.java:165`）把 `parentLocation` 解析成实际的 `BlockModel` 对象，形成继承链。`getElements()` 会回溯到 parent；纹理引用 `getMaterial(name)`（`BlockModel.java:259`）沿 parent 链查找，避免循环。

#### 1.1.3 BlockElement（一个长方体）

`BlockElement`（`BlockElement.java:23`）描述一个 6 面长方体：

```java
public final Vector3f from;        // 起点角（像素，1.21.1 用 JOML Vector3f）
public final Vector3f to;          // 终点角
public final Map<Direction, BlockElementFace> faces;  // 各朝向的面
public final BlockElementRotation rotation;           // 元素级旋转（绕轴+原点）
public final boolean shade;        // 是否参与面方向着色
```

坐标范围允许 `[-16, 32]`（`MIN_EXTENT`/`MAX_EXTENT`，`BlockElement.java:25-26`），即可以超出方块边界。`BlockElement.Deserializer`（`BlockElement.java:77`）解析 `from`/`to` 数组与 `faces` 对象。

#### 1.1.4 BlockElementFace 与 BlockFaceUV（一个面）

`BlockElementFace`（`BlockElementFace.java:16`）是一个 `record`：

```java
public record BlockElementFace(
    @Nullable Direction cullForDirection,  // 剔除面（与邻居实心面贴合则被剔除）
    int tintIndex,                          // 染色索引（-1=NO_TINT）
    String texture,                         // 纹理变量名（指向 textureMap，通常以 # 开头）
    BlockFaceUV uv,                         // UV 坐标
    ...
)
```

`BlockFaceUV`（`BlockFaceUV.java:16`）：

```java
public float[] uvs;      // [u0, v0, u1, v1] 四个角，单位为 1/16 像素
public final int rotation; // UV 旋转（仅 0/90/180/270）
```

`getU(index)`/`getV(index)`（`BlockFaceUV.java:25/34`）根据顶点序号 + rotation 计算实际 UV；`getReverseIndex` 用于 UV 锁定时的反向映射。

**数据流小结（定义层）**：`JSON → BlockModel(parent+elements+textures+transforms) → List<BlockElement(from/to/faces)> → Map<Direction, BlockElementFace(cull/tint/tex/uv)>`。这是一棵"模型 → 元素 → 面"的三层结构，全部在 CPU 内存中，尚未生成任何顶点数据。

### 1.2 模型烘焙

#### 1.2.1 从 BlockModel 到 BakedModel

`BlockModel.bake(...)`（`BlockModel.java:210`）是烘焙入口。原版的 `bakeVanilla`（`BlockModel.java:220`）流程：

1. 取粒子纹理：`spriteGetter.apply(this.getMaterial("particle"))`。
2. 若根模型是 `ModelBakery.BLOCK_ENTITY_MARKER`，返回 `BuiltInModel`（特殊内置模型）。
3. 否则构造 `SimpleBakedModel.Builder`，遍历所有 `BlockElement` 的所有 face：
   - 解析该 face 的纹理：`spriteGetter.apply(this.getMaterial(blockelementface.texture()))`。
   - 调用 `BlockModel.bakeFace`（`BlockModel.java:249`）→ `FACE_BAKERY.bakeQuad(from, to, face, sprite, direction, state, rotation, shade)`。
   - 若 `cullForDirection != null`，用 `addCulledFace`（按朝向分组）；否则 `addUnculledFace`。
4. `builder.build()` 返回 `SimpleBakedModel`。

> 实际上 NeoForge 下 `BlockModel.bake` 被重定向到 `UnbakedGeometryHelper.bake`，但原版 `bakeVanilla` 路径仍是核心参考。

#### 1.2.2 SimpleBakedModel（烘焙结果容器）

`SimpleBakedModel`（`SimpleBakedModel.java:20`）实现了 `BakedModel`：

```java
public final List<BakedQuad> unculledFaces;              // 无剔除面
public final Map<Direction, List<BakedQuad>> culledFaces; // 各朝向剔除面
protected final boolean hasAmbientOcclusion;
protected final TextureAtlasSprite particleIcon;
protected final ItemTransforms transforms;
protected final ItemOverrides overrides;
```

`getQuads(state, direction, random)`（`SimpleBakedModel.java:78`）：`direction == null` 时返回 `unculledFaces`，否则返回 `culledFaces.get(direction)`。

**`BlockModelShaper`**（`BlockModelShaper.java:17`）维护 `Map<BlockState, BakedModel> modelByStateCache`，在资源重载时由 `ModelManager` 重建。`getBlockModel(state)`（`BlockModelShaper.java:36`）从这个缓存取烘焙模型，缺失则用 missing model。

#### 1.2.3 FaceBakery.bakeQuad（核心烘焙）

`FaceBakery`（`FaceBakery.java:21`）持有一个静态实例 `FACE_BAKERY`（`BlockModel.java:52`）。关键常量：`VERTEX_INT_SIZE = 8`（每顶点 8 个 int）、`VERTEX_COUNT = 4`。

`bakeQuad`（`FaceBakery.java:29`）步骤：

1. **UV 锁定**：若 `transform.isUvLocked()`，调用 `recomputeUVs`（`FaceBakery.java:70`）用 `BlockMath.getUVLockTransform` 重新计算 UV，使模型旋转后贴图不跟着转。
2. **UV 收缩抗锯齿**：用 `sprite.uvShrinkRatio()` 把 UV 向中心收紧一点（`FaceBakery.java:46-52`），避免纹理图集采样溢出到相邻 sprite。
3. **生成顶点**：`makeVertices`（`FaceBakery.java:109`）分配 `int[32]`（4 顶点 × 8 int），对每个顶点调 `bakeVertex`。
4. **计算朝向**：`calculateFacing(aint)`（`FaceBakery.java:54`）从顶点几何反推 `Direction`。
5. **重算绕序**：若无元素旋转，`recalculateWinding` 校正面顶点绕序（`FaceBakery.java:57`）。
6. **填充法线**：`ClientHooks.fillNormal(aint, direction)`（NeoForge 钩子，原版逻辑是把面法线 pack 到每个顶点的 Normal 字段）。
7. 构造 `BakedQuad`。

`bakeVertex`（`FaceBakery.java:138`）逐顶点：
- 从 `FaceInfo.fromFacing(facing).getVertexInfo(vertexIndex)` 取该面该顶点对应的角点索引。
- 取 `posDiv16`（即 from/to 除以 16）对应分量，构造 `Vector3f`。
- `applyElementRotation`（元素级旋转，`FaceBakery.java:166`）。
- `applyModelRotation`（模型级 `ModelState` 变换，`FaceBakery.java:204`）。
- `fillVertex`（`FaceBakery.java:156`）写 8 个 int：

```java
vertexData[i]   = floatToRawIntBits(vector.x());  // Position.x
vertexData[i+1] = floatToRawIntBits(vector.y());  // Position.y
vertexData[i+2] = floatToRawIntBits(vector.z());  // Position.z
vertexData[i+3] = -1;                              // Color（ARGB 白色不透明）
vertexData[i+4] = floatToRawIntBits(sprite.getU(u/16));  // UV0.u
vertexData[i+5] = floatToRawIntBits(sprite.getV(v/16));  // UV0.v
// i+6, i+7 对应 UV2(lightmap)/Normal，由 fillNormal 与后续着色阶段填
```

**每顶点 8 int = 32 字节，与 `DefaultVertexFormat.BLOCK` 完全一致**（见 §3.2）。

#### 1.2.4 BakedQuad 结构

`BakedQuad`（`BakedQuad.java:9`）是不可变数据包：

```java
protected final int[] vertices;          // 4 顶点 × 8 int = 32 int
protected final int tintIndex;           // 染色索引（-1 不染色）
protected final Direction direction;     // 面朝向
protected final TextureAtlasSprite sprite;
protected final boolean shade;
protected final boolean hasAmbientOcclusion;
```

顶点数组以 `int` 形式紧凑存储，渲染时通过 `putBulkData` 一次性解码到 `VertexConsumer`。

#### 1.2.5 ModelBlockRenderer（面 → 顶点流）

`ModelBlockRenderer`（`ModelBlockRenderer.java:31`）持有 `BlockColors`，负责把 `BakedModel` 的 quad 写入 `VertexConsumer`。两条主路径：

**区块路径（带光照）** —— `tesselateBlock`（`ModelBlockRenderer.java:74`）：
1. 判定是否启用 AO（`model.useAmbientOcclusion(...)`，结合全局设置与方块发光）。
2. 平移方块 offset（`state.getOffset`，如沙子下沉）。
3. 分派到 `tesselateWithAO` 或 `tesselateWithoutAO`。

`tesselateWithAO`（`ModelBlockRenderer.java:142`）对每个 `Direction`：
- `model.getQuads(state, direction, random, modelData, renderType)` 取该方向 quad。
- 若 `checkSides`，用 `Block.shouldRenderFace` 做面剔除。
- `renderModelFaceAO`（`ModelBlockRenderer.java:256`）：计算面形状 → `AmbientOcclusionFace.calculate` 算出 4 个顶点的亮度/光照图 → `putQuadData`。

`putQuadData`（`ModelBlockRenderer.java:292`）是关键写入点：
```java
if (quad.isTinted()) {
    int i = this.blockColors.getColor(state, level, pos, quad.getTintIndex());
    // 取出 tint 颜色的 r/g/b
}
consumer.putBulkData(pose, quad,
    new float[]{brightness0..3},  // AO 亮度
    r, g, b, 1.0F,
    new int[]{lightmap0..3},       // 光照图
    packedOverlay, true);
```

**实体/物品路径（无场景光照）** —— `renderModel`（`ModelBlockRenderer.java:466`）：
```java
for (Direction direction : DIRECTIONS) {
    random.setSeed(42L);
    renderQuadList(pose, consumer, r, g, b,
        model.getQuads(state, direction, random, modelData, renderType),
        packedLight, packedOverlay);
}
// 再渲染无方向 quad
```
`renderQuadList`（`ModelBlockRenderer.java:491`）对每个 quad 调 `consumer.putBulkData(pose, bakedquad, r, g, b, 1.0F, packedLight, packedOverlay)`，不做 AO 计算。

`VertexConsumer.putBulkData`（`VertexConsumer.java:88`）的逻辑：用 `MemoryStack` 把 quad 的 8 int/顶点解包，对每个顶点用 `pose.pose()` 变换 position、用 `pose.transformNormal` 变换法线，再调 `addVertex/setColor/setUv/setOverlay/setLight/setNormal` 写入。

### 1.3 区块网格化

1.21.1 把原 1.20 的 `ChunkRenderDispatcher` 重命名为 **`SectionRenderDispatcher`**（`net.minecraft.client.renderer.chunk.SectionRenderDispatcher`），并把实际的网格化逻辑抽到 **`SectionCompiler`**。一个"区块渲染单元"现在是 16×16×16 的 section（`RenderSection.SIZE = 16`）。

#### 1.3.1 SectionRenderDispatcher（调度器）

`SectionRenderDispatcher`（`SectionRenderDispatcher.java:54`）是线程池调度器：
- 两个优先级队列：`toBatchHighPriority`（`PriorityBlockingQueue`）、`toBatchLowPriority`。
- `bufferPool`（`SectionBufferBuilderPool`）管理一组可借出的 `SectionBufferBuilderPack`。
- `mailbox`（`ProcessorMailbox`）串行化任务派发。
- `sectionCompiler`（`SectionCompiler`）真正做编译。

**重建流程**（`RenderSection.rebuildSectionAsync`，`SectionRenderDispatcher.java:457`）：
1. `createCompileTask`（`:442`）创建 `RebuildTask`，先用 `RenderRegionCache.createRegion` 构造一个 3×3 chunk 的 `RenderChunkRegion`（带邻居用于面剔除）。
2. `sectionRenderDispatcher.schedule(task)` 入队。
3. 工作线程取任务 → `RebuildTask.doTask(sectionBufferBuilderPack)`（`:546`）→ `sectionCompiler.compile(...)`。
4. 编译产出 `SectionCompiler.Results`（含 `Map<RenderType, MeshData>`）。
5. 对每层调 `uploadSectionLayer(meshData, vertexBuffer)`（`:203`）在渲染线程上传到 `VertexBuffer`。
6. `RenderSection.setCompiled(new CompiledSection)`。

#### 1.3.2 SectionCompiler.compile（核心网格化）

`SectionCompiler.compile`（`SectionCompiler.java:47`）：

```java
Map<RenderType, BufferBuilder> map = new Reference2ObjectArrayMap<>(RenderType.chunkBufferLayers().size());
PoseStack posestack = new PoseStack();
VisGraph visgraph = new VisGraph();

for (BlockPos blockpos2 : BlockPos.betweenClosed(blockpos, blockpos1)) {  // 遍历 4096 个方块
    BlockState blockstate = region.getBlockState(blockpos2);
    if (blockstate.isSolidRender(...)) visgraph.setOpaque(blockpos2);  // 记录不透明方块（用于遮挡图）
    if (blockstate.hasBlockEntity()) handleBlockEntity(...);            // 收集 BlockEntity
    FluidState fluidstate = blockstate.getFluidState();
    if (!fluidstate.isEmpty()) {
        RenderType rt = ItemBlockRenderTypes.getRenderLayer(fluidstate);
        BufferBuilder bb = getOrBeginLayer(map, pack, rt);
        blockRenderer.renderLiquid(...);                                // 流体
    }
    if (blockstate.getRenderShape() == RenderShape.MODEL) {
        for (RenderType rt2 : model.getRenderTypes(state, random, modelData)) {
            BufferBuilder bb1 = getOrBeginLayer(map, pack, rt2);
            posestack.pushPose();
            posestack.translate(sectionRelative(x), sectionRelative(y), sectionRelative(z));  // 局部坐标 0..16
            blockRenderer.renderBatched(state, pos, region, posestack, bb1, true, random, modelData, rt2);
            posestack.popPose();
        }
    }
}
// build 所有层
for (Entry<RenderType, BufferBuilder> e : map.entrySet()) {
    MeshData meshdata = e.getValue().build();
    if (meshdata != null) {
        if (e.getKey() == RenderType.translucent())
            results.transparencyState = meshdata.sortQuads(...);  // 半透明排序
        results.renderedLayers.put(e.getKey(), meshdata);
    }
}
results.visibilitySet = visgraph.resolve();  // 邻接可见性
```

`getOrBeginLayer`（`SectionCompiler.java:118`）：按 `RenderType` 从 `SectionBufferBuilderPack.buffer(rt)` 取 `ByteBufferBuilder`，包装成 `BufferBuilder(bb, VertexFormat.Mode.QUADS, DefaultVertexFormat.BLOCK)`。

**`blockRenderer.renderBatched`**（`BlockRenderDispatcher.java:74`）→ `modelRenderer.tesselateBlock(...)`（见 §1.2.5）。这里 `checkSides=true` 启用面剔除，顶点坐标是 section 局部坐标（0..16）。

#### 1.3.3 RenderType → BufferBuilder 的分配

区块使用 `RenderType.chunkBufferLayers()`（`RenderType.java:1154`），返回固定的 5 层：

```java
public static final ImmutableList<RenderType> CHUNK_BUFFER_LAYERS =
    ImmutableList.of(solid(), cutoutMipped(), cutout(), translucent(), tripwire());
```

每个 `RenderSection` 为这 5 层各持有一个 `VertexBuffer`：

```java
private final Map<RenderType, VertexBuffer> buffers = RenderType.chunkBufferLayers().stream()
    .collect(... -> new VertexBuffer(VertexBuffer.Usage.STATIC));   // SectionRenderDispatcher.java:305
```

方块归属哪一层由 `ItemBlockRenderTypes` 决定（见 §1.5）。

#### 1.3.4 SectionBufferBuilderPack（缓冲池）

`SectionBufferBuilderPack`（`SectionBufferBuilderPack.java:12`）为每个 chunk layer 预分配一个 `ByteBufferBuilder`，大小取自 `RenderType.bufferSize()`（如 solid/cutoutMipped 是 4MB，cutout/translucent 是 786KB）。`SectionBufferBuilderPool` 持有多个 pack，编译任务借用/归还。

#### 1.3.5 CompiledSection 与可见性

`CompiledSection`（`SectionRenderDispatcher.java:256`）保存编译结果：
- `hasBlocks`：`Set<RenderType>`（哪些层非空）。
- `renderableBlockEntities`：需要单独渲染的 BlockEntity 列表。
- `visibilitySet`：`VisibilitySet`，由 `VisGraph.resolve()` 生成 —— 标记任意两个方向之间是否"视线可穿透"（用于 section 级遮挡剔除）。
- `transparencyState`：`MeshData.SortState`，半透明 quad 的质心，用于相机移动时重排序。

`RenderSection`（`SectionRenderDispatcher.java:293`）是渲染时的单元：`origin`（section 原点世界坐标）、`compiled`（`AtomicReference<CompiledSection>`）、`buffers`（5 个 `VertexBuffer`）。`getBuffer(renderType)`（`:343`）返回对应层缓冲。

#### 1.3.6 数据源：RenderChunkRegion 与 RenderChunk

注意 `RenderChunk`（`RenderChunk.java:24`，**包级私有**）与 `RenderSection`（渲染单元）是两个不同的类：
- `RenderChunk` 包装一个 `LevelChunk`，并在构造时**快照**它的 `PalettedContainer<BlockState>` sections（`RenderChunk.java:38-44`），保证多线程编译期间读到一致状态。
- `RenderChunkRegion`（`RenderChunkRegion.java:18`）持有 3×3 的 `RenderChunk[]`（`minChunkX/minChunkZ` 起），实现 `BlockAndTintGetter`，供 `SectionCompiler` 查询方块状态/流体/BlockEntity/ModelData。

`RenderRegionCache`（`:3.1KB`）负责构建这个 region（带邻居以便边界方块做正确的面剔除）。

#### 1.3.7 上传与透明排序

`uploadSectionLayer`（`SectionRenderDispatcher.java:203`）在渲染线程把 `MeshData` 上传到 `VertexBuffer`：
```java
vertexBuffer.bind();
vertexBuffer.upload(meshData);
VertexBuffer.unbind();
```

半透明层（`RenderType.translucent()`）额外维护 `ResortTransparencyTask`（`SectionRenderDispatcher.java:607`）：当相机跨 section 边界时，用 `MeshData.SortState.buildSortedIndexBuffer` 重算索引顺序并 `uploadSectionIndexBuffer`。

### 1.4 流体渲染

`LiquidBlockRenderer`（`LiquidBlockRenderer.java:29`）专门处理水/岩浆等流体。在 `SectionCompiler` 中，流体的 `RenderType` 由 `ItemBlockRenderTypes.getRenderLayer(fluidState)` 决定（水和岩浆都是 `translucent`），然后：

```java
blockRenderer.renderLiquid(blockpos2, region, bufferbuilder, blockstate, fluidstate);
```

`BlockRenderDispatcher.renderLiquid`（`BlockRenderDispatcher.java:109`）→ `liquidBlockRenderer.tesselate(level, pos, consumer, blockState, fluidState)`。

`LiquidBlockRenderer.tesselate`（`LiquidBlockRenderer.java:87`）流程：
1. 取流体纹理（`FluidSpriteCache.getFluidSprites`，返回 `[still, flow, overlay]`）与染色（`IClientFluidTypeExtensions.getTintColor`）。
2. 检查 6 个邻居是否同流体 / 是否遮挡，决定哪些面要渲染（`shouldRenderFace`）。
3. 计算流体表面高度：`getHeight`（`LiquidBlockRenderer.java:341`）查 4 个角相邻方块，得到 0..1 的液面高度（`MAX_FLUID_HEIGHT = 0.8888889F`）。
4. 渲染顶面、底面、四个侧面，每个面是一个 quad，调用 `vertex(...)` 写入顶点。

`vertex`（`LiquidBlockRenderer.java:346`）通过 `VertexConsumer` 链式 API 写入：
```java
buffer.addVertex(x, y, z)
```

## 2. 实体几何管线

### 2.1 模型定义层

实体模型是**编程式定义**的（不是 JSON），通过 builder API 在代码里搭建 `ModelPart` 树。

#### 2.1.1 注册中心：LayerDefinitions 与 EntityModelSet

`LayerDefinitions`（`LayerDefinitions.java:114`）的 `createRoots()`（`LayerDefinitions.java:119`）返回 `Map<ModelLayerLocation, LayerDefinition>`，在资源重载时被调用。它把每个实体的模型层（如 `ModelLayers.PLAYER`、`ModelLayers.CREEPER`）映射到一个 `LayerDefinition`：

```java
LayerDefinition layerdefinition = LayerDefinition.create(
    HumanoidModel.createMesh(CubeDeformation.NONE, 0.0F), 64, 64);
builder.put(ModelLayers.PLAYER, LayerDefinition.create(PlayerModel.createMesh(...), 64, 64));
builder.put(ModelLayers.CREEPER, CreeperModel.createBodyLayer(CubeDeformation.NONE));
...
```

`ModelLayers`（`ModelLayers.java`）定义所有 `ModelLayerLocation` 常量（实体主模型 + 装甲层 + 外层如羊毛/伤疤）。

`EntityModelSet`（`EntityModelSet.java:12`）实现 `ResourceManagerReloadListener`，在 `onResourceManagerReload`（`:25`）时缓存 `LayerDefinitions.createRoots()` 的结果。`bakeLayer(ModelLayerLocation)`（`:15`）调 `LayerDefinition.bakeRoot()` 产出 `ModelPart`。

#### 2.1.2 构建器：MeshDefinition / PartDefinition / CubeListBuilder

`LayerDefinition`（`LayerDefinition.java:8`）封装两部分：
- `MeshDefinition`（`MeshDefinition.java:9`）：持有一个根 `PartDefinition`。
- `MaterialDefinition`：纹理宽高（如 64×64）。

`PartDefinition`（`PartDefinition.java:16`）：
```java

private final PartPose partPose;
private final Map<String, PartDefinition> children;
```
`addOrReplaceChild(name, CubeListBuilder, PartPose)`（`:26`）递归构建子部件树。

`bake(texWidth, texHeight)`（`PartDefinition.java:36`）把定义树转成 `ModelPart` 树：递归 bake children，每个 `CubeDefinition.bake` 产出 `ModelPart.Cube`，最终 `new ModelPart(cubes, childrenMap)` 并 `loadPose(partPose)`。

#### 2.1.3 ModelPart（运行时模型节点）

`ModelPart`（`ModelPart.java:19`）是运行时的实体模型节点：

```java
public float x, y, z;              // 相对父节点的位移（1/16 像素单位）
public float xRot, yRot, zRot;     // 欧拉角旋转
public float xScale, yScale, zScale = 1.0F;
public boolean visible = true;
public final List<ModelPart.Cube> cubes;
public final Map<String, ModelPart> children;
private PartPose initialPose;
```

它是一棵树：每个节点有自己的 cubes 和 children。`getChild(name)`（`:85`）按名取子节点（如 `"head"`、`"right_arm"`）。

**`render`（渲染入口）** —— `ModelPart.render`（`ModelPart.java:110`）：
```java
if (this.visible) {
    poseStack.pushPose();
    this.translateAndRotate(poseStack);              // 应用本节点 pose
    if (!this.skipDraw) this.compile(poseStack.last(), buffer, packedLight, packedOverlay, color);  // 画 cubes
    for (ModelPart child : this.children.values())    // 递归子节点
        child.render(poseStack, buffer, packedLight, packedOverlay, color);
    poseStack.popPose();
}
```

`translateAndRotate`（`ModelPart.java:148`）：
```java
poseStack.translate(x/16, y/16, z/16);
if (有旋转) poseStack.mulPose(new Quaternionf().rotationZYX(zRot, yRot, xRot));  // ZYX 顺序
if (有缩放) poseStack.scale(xScale, yScale, zScale);
```

#### 2.1.4 ModelPart.Cube（长方体几何）

`ModelPart.Cube`（`ModelPart.java:196`）在**构造时**就把 8 个角点和最多 6 个 `Polygon` 全部算好（不同于方块的运行时烘焙，实体模型几何是 bake-once 的）：

```java
public final ModelPart.Polygon[] polygons;
public final float minX, minY, minZ, maxX, maxY, maxZ;
```

构造器（`ModelPart.java:205`）接收纹理原点 `(texU, texV)`、尺寸、grow（外扩）、mirror、`Set<Direction> visibleFaces`，按经典 cube UV 展开公式计算 6 个面的纹理坐标，为每个可见方向构造一个 `Polygon`（含 4 个 `Vertex`）。

`compile`（`ModelPart.java:347`）是实际的顶点写入：
```java
for (ModelPart.Polygon polygon : this.polygons) {
    Vector3f normal = pose.transformNormal(polygon.normal, ...);  // 法线变换
    for (ModelPart.Vertex v : polygon.vertices) {
        Vector3f pos = matrix4f.transformPosition(v.pos.x/16, v.pos.y/16, v.pos.z/16, ...);
        buffer.addVertex(pos.x, pos.y, pos.z, color, v.u, v.v, packedOverlay, packedLight, normal.x, normal.y, normal.z);
    }
}
```
注意 position/normal 都通过 `PoseStack.Pose` 当前的矩阵变换，UV 直接来自 bake 时算好的值。

#### 2.1.5 ModelPart.Polygon / Vertex

- `Polygon`（`ModelPart.java:371`）：`Vertex[4]` + `Vector3f normal`（= `direction.step()`，mirror 时 x 取反）。
- `Vertex`（`ModelPart.java:411`）：`Vector3f pos` + `float u, v`。

#### 2.1.6 EntityModel 抽象

`EntityModel<T extends Entity>`（`EntityModel.java:11`）继承 `Model`，增加：
- `attackTime`、`riding`、`young` 字段。
- 抽象方法 `setupAnim(entity, limbSwing, limbSwingAmount, ageInTicks, netHeadYaw, headPitch)` —— 子类在此驱动各 `ModelPart` 的旋转（动画）。
- `prepareMobModel(...)` 预备动画。

`Model`（`Model.java:12`）持有一个 `Function<ResourceLocation, RenderType> renderType`（默认 `RenderType::entityCutoutNoCull`），抽象方法 `renderToBuffer(poseStack, buffer, packedLight, packedOverlay, color)`。

**数据流小结（定义层）**：`LayerDefinitions.createRoots() → Map<ModelLayerLocation, LayerDefinition> → EntityModelSet.bakeLayer → LayerDefinition.bakeRoot → PartDefinition.bake → ModelPart 树(cubes+children) → ModelPart.Cube(polygons+vertices)`。

### 2.2 渲染流程

#### 2.2.1 LivingEntityRenderer.render（主流程）

`LivingEntityRenderer`（`LivingEntityRenderer.java:31`）是所有生物渲染器的基类，持有 `M model`（`EntityModel`）和 `List<RenderLayer>`。

`render`（`LivingEntityRenderer.java:52`）流程：

1. **pose 准备**：`pushPose`，计算身体朝向 `f`（`yBodyRot` 插值）、头部朝向 `f1`（`yHeadRot` 插值）、俯仰 `f6`（`xRot` 插值）。
2. **整体变换**（`setupRotations`，`:181`）：`scale(entityScale)`、绕 Y 轴转 `180 - yBodyRot`、死亡/睡觉/旋转攻击等特殊旋转、`scale(-1,-1,1)`（模型坐标系翻转）、`translate(0,-1.501,0)`（把脚放到原点）。
3. **动画**：`model.prepareMobModel(...)` + `model.setupAnim(entity, limbSwing, limbSwingAmount, ageInTicks, netHeadYaw, headPitch)` —— 子类在此设置每个 `ModelPart` 的旋转角。
4. **选 RenderType**：`getRenderType`（`:143`）根据可见性/透明/发光返回 `RenderType`（如 `entityCutoutNoCull(texture)`、`itemEntityTranslucentCull`、`outline`）。
5. **写入模型**：
   ```java
   VertexConsumer vertexconsumer = buffer.getBuffer(rendertype);
   this.model.renderToBuffer(poseStack, vertexconsumer, packedLight, overlay, color);
   ```
   `renderToBuffer`（由 `EntityModel` 子类实现）通常直接调 `rootModel.render(poseStack, buffer, ...)`。
6. **叠加层**：
   ```java
   for (RenderLayer<T,M> renderlayer : this.layers)
       renderlayer.render(poseStack, buffer, packedLight, entity, ...);
   ```
7. `popPose`，调 `super.render`（`EntityRenderer.render`，通常是空或标签）。

#### 2.2.2 矩阵变换链

实体渲染的矩阵栈（`PoseStack`）：
- 由 `EntityRenderDispatcher.render`（`EntityRenderDispatcher.java:142`）在调用前 `pushPose` + `translate(entityX, entityY, entityZ)`（已减去相机位置）。
- `LivingEntityRenderer` 再叠加身体朝向、缩放、模型坐标系翻转。
- `ModelPart.render` 递归地 `pushPose`/`translateAndRotate`，每个骨骼节点的局部变换层层累加。
- `Cube.compile` 用最终的 `pose.pose()`（`Matrix4f`）变换顶点位置，用 `pose.transformNormal` 变换法线。

#### 2.2.3 顶点写入

实体模型使用 `DefaultVertexFormat.NEW_ENTITY`（见 §3.2），顶点写入通过 `VertexConsumer.addVertex(x,y,z,color,u,v,overlay,light,nx,ny,nz)`（`VertexConsumer.java:28` 的 default 方法）：
```java
this.addVertex(x, y, z);
this.setColor(color);
this.setUv(u, v);
this.setOverlay(packedOverlay);   // UV1
this.setLight(packedLight);        // UV2
this.setNormal(nx, ny, nz);
```

#### 2.2.4 EntityRenderDispatcher（派发）

`LevelRenderer.renderEntity`（`LevelRenderer.java:1259`）→ `entityRenderDispatcher.render(entity, x, y, z, yaw, partialTick, poseStack, buffer, packedLight)`（`EntityRenderDispatcher.java:142`）：
1. 取 `EntityRenderer`。
2. `pushPose` + `translate`（含 `getRenderOffset`）。
3. `entityrenderer.render(...)`（对生物即 `LivingEntityRenderer.render`）。
4. 火焰动画、阴影（`renderShadow`）、命中盒（debug）。
5. `popPose`。

### 2.3 渲染层叠加

`RenderLayer<T extends Entity, M extends EntityModel<T>>`（`RenderLayer.java:17`，注意这是 `net.minecraft.client.renderer.entity.layers.RenderLayer`，与 §3.1 的 `RenderType` 不同）是实体"附加层"的基类，如装甲、鞘翅、羊毛、伤疤覆盖、僵尸头部等。

- 通过 `LivingEntityRenderer.addLayer(...)`（`LivingEntityRenderer.java:43`）注册。
- 抽象方法 `render(poseStack, buffer, packedLight, entity, limbSwing, ...)`（`RenderLayer.java:63`）。
- 工具方法 `coloredCutoutModelCopyLayerRender`（`:24`）和 `renderColoredCutoutModel`（`:48`）：复用父模型骨骼 pose，换一张纹理/换一个 `EntityModel` 再画一遍：
  ```java

  model.renderToBuffer(poseStack, vc, packedLight, overlay, color);
  ```

典型叠加层（`net.minecraft.client.renderer.entity.layers/` 下）：`HumanoidArmorLayer`（装甲）、`ElytraLayer`、`CapeLayer`、`CustomHeadLayer`、`SheepFurLayer`、`StrayClothingLayer` 等。它们共享父渲染器的 `ModelPart` pose（动画同步），但各自取不同的纹理和 RenderType。

### 2.4 BlockEntity 渲染（简要）

BlockEntity（如箱子、信标、末影箱、告示牌）由 `BlockEntityRenderer` 单独渲染，不进 chunk 顶点缓冲。

`BlockEntityRenderDispatcher`（`BlockEntityRenderDispatcher.java:30`）维护 `Map<BlockEntityType<?>, BlockEntityRenderer<?>>`（在 `onResourceManagerReload` 时由 `BlockEntityRenderers.createEntityRenderers` 构造）。

渲染流程（`BlockEntityRenderDispatcher.java:69`）：
1. `getRenderer(blockEntity)` 取对应渲染器。
2. `shouldRender` 判定（含离屏可见性）。
3. `setupAndRender`（`:80`）：取方块光照 `LevelRenderer.getLightColor`，调 `renderer.render(blockEntity, partialTick, poseStack, bufferSource, packedLight, NO_OVERLAY)`。

每个 `BlockEntityRenderer` 实现自己的 `render`，通常用 `EntityModelSet` 取预烘焙的 `ModelPart`（如 `ChestRenderer` 用 `ModelLayers.CHEST`），或直接调 `BlockRenderDispatcher.renderSingleBlock` 画方块模型。

在 `SectionCompiler` 中，带 `BlockEntityRenderer` 的方块被收集进 `CompiledSection.renderableBlockEntities`（`SectionCompiler.handleBlockEntity`，`SectionCompiler.java:129`），然后在 `LevelRenderer` 的 `blockentities` 阶段逐个渲染。

## 3. 共享渲染基础设施

### 3.1 RenderType 系统

#### 3.1.1 RenderType 与 RenderStateShard

`RenderType`（`RenderType.java:21`）继承自 `RenderStateShard`，是"如何绘制一段顶点数据"的完整描述。每个 `RenderType` 实例携带：
- `format`：`VertexFormat`（顶点布局）。
- `mode`：`VertexFormat.Mode`（`QUADS`/`LINES`/`TRIANGLES`...）。
- `bufferSize`：建议缓冲大小。
- `affectsCrumbling`：是否影响破坏进度渲染。
- `sortOnUpload`：上传前是否按相机距离排序（仅 `translucent` 为 true）。

`CompositeRenderType`（`RenderType.java:1191`）是唯一的具体子类，持有一个 `CompositeState`（`RenderType.java:1258`），后者是 13 个 **`RenderStateShard`** 的组合：

| StateShard | 作用 |
|-----------|------|
| `TextureStateShard` | 绑定纹理 + 设置 blur/mipmap（`RenderStateShard.java:612`） |
| `ShaderStateShard` | `RenderSystem.setShader(supplier)`（`:590`） |
| `TransparencyStateShard` | 混合函数（`NO_TRANSPARENCY`/`TRANSLUCENT_TRANSPARENCY`/...，`:28`起） |
| `DepthTestStateShard` | 深度测试 |
| `CullStateShard` | 面剔除 |
| `LightmapStateShard` | 光照图开关 |
| `OverlayStateShard` | overlay（伤害红闪）开关 |
| `LayeringStateShard` | 视图偏移 |
| `OutputStateShard` | 渲染目标（`OUTLINE_TARGET`/`PARTICLES_TARGET`/...） |
| `TexturingStateShard` | 纹理坐标特殊处理（附魔光泽） |
| `WriteMaskStateShard` | 颜色/深度写掩码 |
| `LineStateShard` | 线宽 |
| `ColorLogicStateShard` | 逻辑操作 |

每个 shard 有 `setupState`/`clearState` 两个 `Runnable`。`CompositeRenderType` 构造时（`:1227`）把它们组合成"setup = 遍历调每个 shard.setupRenderState()"。

#### 3.1.2 RenderType 的创建与绘制

`RenderType.create(name, format, mode, bufferSize, affectsCrumbling, sortOnUpload, CompositeState)`（`RenderType.java:1131`）创建 `CompositeRenderType`。例如（`RenderType.java:26`）：

`draw(MeshData)`（`RenderType.java:1143`）：
```java

BufferUploader.drawWithShader(meshData);
this.clearRenderState();             // 还原
```

#### 3.1.3 常用 RenderType

- **区块层**（`RenderType.java:745`）：`solid`、`cutoutMipped`、`cutout`、`translucent`、`tripwire`，都用 `DefaultVertexFormat.BLOCK`，纹理为 `TextureAtlas.LOCATION_BLOCKS`。
- **实体**（`:74`起）：`entitySolid(rl)`、`entityCutout(rl)`、`entityCutoutNoCull(rl)`、`entityTranslucent(rl)`、`armorCutoutNoCull(rl)` 等，用 `DefaultVertexFormat.NEW_ENTITY`，带 overlay+lightmap，按 `ResourceLocation` memoize。
- **其他**：`lines()`、`lightning()`、`outline(rl)`、`gui()`、`text(rl)`、`particles`、`eyes(rl)` 等。

### 3.2 顶点缓冲系统（Blaze3D）

位于 `com/mojang/blaze3d/vertex/`。

#### 3.2.1 VertexFormatElement 与 VertexFormat

`VertexFormatElement`（`VertexFormatElement.java:12`）是 record，定义单个顶点属性：

```java
public static final VertexFormatElement POSITION = register(0, 0, FLOAT, POSITION, 3);  // 3 float
public static final VertexFormatElement COLOR    = register(1, 0, UBYTE, COLOR, 4);     // 4 ubyte
public static final VertexFormatElement UV0      = register(2, 0, FLOAT, UV, 2);        // 2 float (纹理)
public static final VertexFormatElement UV1      = register(3, 1, SHORT, UV, 2);        // 2 short (overlay)
public static final VertexFormatElement UV2      = register(4, 2, SHORT, UV, 2);        // 2 short (lightmap)
public static final VertexFormatElement NORMAL   = register(5, 0, BYTE, NORMAL, 3);     // 3 byte
```

`VertexFormat`（`VertexFormat.java:16`）通过 `builder()`（`:39`）组装元素列表，算出 `vertexSize` 与各元素偏移。

#### 3.2.2 DefaultVertexFormat（关键布局）

`DefaultVertexFormat`（`DefaultVertexFormat.java:7`）定义所有内置布局：

**`BLOCK`**（32 字节，方块/流体用）：
```
Position(3f=12) | Color(4ub=4) | UV0(2f=8) | UV2(2short=4) | Normal(3byte=3) | padding(1)
```
注意 BLOCK **没有 UV1(overlay)** —— 方块不需要受伤害红闪。这与 `BakedQuad` 的 8 int/顶点完全对应。

**`NEW_ENTITY`**（36 字节，实体用）：
```
Position(3f=12) | Color(4ub=4) | UV0(2f=8) | UV1(2short=4) | UV2(2short=4) | Normal(3byte=3) | padding(1)
```
比 BLOCK 多了 UV1(overlay)，用于实体的伤害覆盖层与手持物品高光。

其他：`PARTICLE`、`POSITION_COLOR_NORMAL`、`POSITION_TEX_COLOR_NORMAL` 等用于粒子、线、调试框等。

#### 3.2.3 VertexConsumer 接口

`VertexConsumer`（`VertexConsumer.java:15`）是顶点写入的统一抽象（继承 NeoForge 的 `IVertexConsumerExtension`）。核心方法：`addVertex(x,y,z)`、`setColor`、`setUv`、`setUv1`、`setUv2`、`setNormal`，全部返回 `VertexConsumer` 支持链式调用。

default 方法 `addVertex(x,y,z,color,u,v,packedOverlay,packedLight,nx,ny,nz)`（`:28`）封装了一次完整的 11 属性写入（实体用）。`setOverlay`/`setLight`（`:67`/`:63`）把 packed 值拆成 `(u,v)` 写入 UV1/UV2。

`putBulkData`（`:71`/`:88`）是方块专用的高效批量写入：直接从 `BakedQuad.getVertices()` 的 int 数组解包，用 `PoseStack.Pose` 变换后写入，避免逐顶点链式调用。

#### 3.2.4 BufferBuilder（实现 VertexConsumer）

`BufferBuilder`（`BufferBuilder.java:13`）是 `VertexConsumer` 的主要实现，持有一个 `ByteBufferBuilder`（堆外内存增长缓冲）。

关键机制（`BufferBuilder.java:88`起）：
- `beginVertex()`：`buffer.reserve(vertexSize)` 申请一顶点空间，返回指针 `vertexPointer`。
- `beginElement(element)`：用位掩码 `elementsToFill` 跟踪当前顶点哪些元素已写，避免重复；返回元素在顶点内的绝对指针。
- `addVertex(x,y,z)`（`:143`）：写 Position，重置 `elementsToFill = initialElementsToFill`（除 Position 外所有元素）。
- `setColor`/`setUv`/`setLight`/`setNormal`（`:152`起）：通过 `beginElement` 取偏移，用 `MemoryUtil.memPutFloat/memPutByte/memPutShort` 直接写裸内存。
- `endLastVertex()`（`:113`）：校验所有元素已写；LINES 模式额外复制顶点。
- `build()`（`:48`）→ `storeMesh()`（`:73`）：把 `ByteBufferBuilder.Result` 连同 `DrawState(format, vertexCount, indexCount, mode, indexType)` 打包成 `MeshData`。

`fastFormat`（`:42`）标记 NEW_ENTITY/BLOCK 格式，启用快速路径。

#### 3.2.5 MeshData（待绘制网格）

`MeshData`（`MeshData.java:14`）是不可变的待绘制数据包：
- `vertexBuffer`：`ByteBufferBuilder.Result`（顶点字节）。
- `indexBuffer`：可选索引字节（半透明排序后用；否则用 `AutoStorageIndexBuffer` 顺序索引）。
- `drawState`：`DrawState(format, vertexCount, indexCount, mode, indexType)`。
- `sortQuads`（`:65`）：计算每个 quad 的质心，返回 `SortState`，用于后续按相机距离重排。
- `SortState`（`:90`）：`Vector3f[] centroids` + `IndexType`，`buildSortedIndexBuffer` 据此生成排序索引。

#### 3.2.6 MultiBufferSource / BufferSource

`MultiBufferSource`（`MultiBufferSource.java:17`）接口只有一个方法 `getBuffer(RenderType)`，返回 `VertexConsumer`。核心实现 `BufferSource`（`:29`）：

- `sharedBuffer`：共享 `ByteBufferBuilder`（互斥的 RenderType 复用）。
- `fixedBuffers`：`SequencedMap<RenderType, ByteBufferBuilder>`（专用缓冲，如区块 5 层）。
- `startedBuilders`：`Map<RenderType, BufferBuilder>` 当前活动的 builder。

`getBuffer(renderType)`（`:42`）：
1. 若已有 builder 且 RenderType 允许合并 → 复用。
2. 否则在 fixedBuffers 或 sharedBuffer 上新建 `BufferBuilder`。
3. 共享缓冲会先把上一个 RenderType flush（因为共用内存）。

`endBatch(renderType)`（`:84`）→ `endBatch(rt, builder)`（`:91`）：
```java
MeshData meshdata = builder.build();
if (meshdata != null) {
    if (renderType.sortOnUpload()) meshdata.sortQuads(sharedBuffer, RenderSystem.getVertexSorting());
    renderType.draw(meshdata);   // 见 §3.1.2
}
```

#### 3.2.7 VertexBuffer（GPU 缓冲对象）

`VertexBuffer`（`VertexBuffer.java:14`）封装 OpenGL 的 VBO + IBO + VAO：
```java
public int vertexBufferId;   // GL_ARRAY_BUFFER
public int indexBufferId;    // GL_ELEMENT_ARRAY_BUFFER
public int arrayObjectId;    // VAO
public VertexFormat format;
public RenderSystem.AutoStorageIndexBuffer sequentialIndices;  // 顺序索引（QUADS 时省略显式索引）
public int indexCount;
public VertexFormat.Mode mode;
```

- `upload(MeshData)`（`:35`）：调 `uploadVertexBuffer`（`:112`，绑定 `GL_ARRAY_BUFFER` 并 `glBufferData`）与 `uploadIndexBuffer`（`:136`）。若 meshData 无显式索引，则用 `RenderSystem.getSequentialBuffer(mode)` 的共享顺序索引。
- `bind()`（`:151`）：`GlStateManager._glBindVertexArray(arrayObjectId)`。
- `draw()`（`:161`）：`RenderSystem.drawElements(mode.asGLMode, indexCount, indexType.asGLType)` —— 真正的 draw call。
- `drawWithShader(modelView, projection, shader)`（`:170`）→ `_drawWithShader`（`:178`）：`shader.setDefaultUniforms(...)` → `shader.apply()` → `draw()` → `shader.clear()`。

`Usage.STATIC`（区块数据，`GL_STATIC_DRAW`）/`DYNAMIC`（即时渲染）。

#### 3.2.8 BufferUploader（即时绘制）

`BufferUploader`（`BufferUploader.java:9`）用于非 chunk 的即时绘制（实体、GUI、粒子）：
- `drawWithShader(MeshData)`（`:24`）→ `_drawWithShader`（`:32`）：
  ```java
  VertexBuffer vb = upload(meshData);   // bindImmediateBuffer(format.getImmediateDrawVertexBuffer())
  vb.drawWithShader(modelView, projection, RenderSystem.getShader());
  ```
- 每个 `VertexFormat` 有一个 `immediateDrawVertexBuffer`（`VertexFormat.java:24`），可重复 upload+draw。
- `lastImmediateBuffer` 避免重复 bind 同一 VAO。

### 3.3 着色器系统

#### 3.3.1 ShaderInstance（着色器程序）

`ShaderInstance`（`ShaderInstance.java:45`）实现 `Shader`，封装一个 GL program：
- `programId`：链接后的 program。
- `vertexProgram` / `fragmentProgram`：`Program` 对象（编译后的 GLSL）。
- `vertexFormat`：该着色器期望的顶点布局。
- `samplerMap`：纹理采样器（`Sampler0..N`）。
- `uniformMap` / `uniforms`：`Uniform` 列表。

着色器从 `shaders/core/*.json` 加载（`SHADER_CORE_PATH`，`ShaderInstance.java:47`），JSON 描述顶点/片元源文件、uniform 默认值、sampler。

#### 3.3.2 Uniform 系统

`ShaderInstance` 暴露一组常用 Uniform 字段（`ShaderInstance.java:67`起），在构造时按名查找：
- `MODEL_VIEW_MATRIX`、`PROJECTION_MATRIX`、`TEXTURE_MATRIX`
- `COLOR_MODULATOR`、`FOG_START/END/COLOR/SHAPE`、`GAME_TIME`、`SCREEN_SIZE`
- `LIGHT0_DIRECTION`、`LIGHT1_DIRECTION`（平行光方向）
- `CHUNK_OFFSET`（区块渲染专用，见 §3.4）
- `GLINT_ALPHA`、`LINE_WIDTH`

`setDefaultUniforms(mode, frustumMatrix, projectionMatrix, window)`（`ShaderInstance.java:431`）：把 ModelView/Projection/Fog/Color/GameTime 等从 `RenderSystem` 全局状态灌进 uniform，并 `setSampler("Sampler"+i, ...)` 绑定纹理。

#### 3.3.3 apply / clear

`apply()`（`ShaderInstance.java:251`）：
1. `ProgramManager.glUseProgram(programId)`（带 lastProgramId 缓存避免重复切换）。
2. 遍历 sampler：`Uniform.uploadInteger(loc, unit)` + `RenderSystem.activeTexture/bindTexture`。
3. 遍历 uniforms：`uniform.upload()`。

`clear()`（`:234`）：`glUseProgram(0)`，解绑 sampler 纹理。

`ShaderStateShard`（`RenderStateShard.java:590`）在 `setupRenderState` 时调 `RenderSystem.setShader(supplier)`，把着色器塞进全局，绘制时由 `RenderSystem.getShader()` 取回。

### 3.4 主渲染循环

#### 3.4.1 GameRenderer.render → renderLevel

`GameRenderer.render(DeltaTracker, renderLevel)`（`GameRenderer.java:1004`）是每帧入口。在 `flag && renderLevel` 时调 `renderLevel(deltaTracker)`（`:1231`），它准备相机矩阵后调：
```java
this.minecraft.levelRenderer.renderLevel(deltaTracker, flag, camera, this, lightTexture, frustumMatrix, projectionMatrix);
```

#### 3.4.2 LevelRenderer.renderLevel（核心渲染顺序）

`LevelRenderer.renderLevel`（`LevelRenderer.java:921`）按固定顺序驱动整帧（行号引自实际源码）：

1. **准备**（`:927-928`）：`blockEntityRenderDispatcher.prepare`、`entityRenderDispatcher.prepare`。
2. **光照更新**（`:931-933`）。
3. **视锥剔除**（`:938-946`）：`Frustum`。
4. **清屏 + 天空**（`:954-963`）：`RenderSystem.clear`、`renderSky`。
5. **雾设置**（`:965-966`）。
6. **地形 setup + 编译**（`:967-970`）：`setupRender`（构建可见 section 列表 `visibleSections`、运行 `SectionOcclusionGraph` 遮挡剔除）、`compileSections`（把脏 section 投递给 `SectionRenderDispatcher`）。
7. **地形绘制（不透明）**（`:971-976`）：
   ```java
   renderSectionLayer(RenderType.solid(), ...);
   // 切换树叶 atlas mipmap
   renderSectionLayer(RenderType.cutoutMipped(), ...);
   renderSectionLayer(RenderType.cutout(), ...);
   ```
8. **光照设置**（`:977-981`）：`Lighting.setupLevel`。
9. **实体**（`:983-1044`）：遍历 `level.entitiesForRendering()`，`entityRenderDispatcher.shouldRender` 剔除后 `renderEntity` → `entityRenderDispatcher.render`（§2.2.4）。实体顶点写入 `MultiBufferSource.BufferSource`。
10. **实体缓冲 flush**（`:1046-1051`）：`endBatch` 各 entity RenderType。
11. **BlockEntity**（`blockentities` 阶段）：渲染 `CompiledSection.renderableBlockEntities` + 全局 BlockEntity。
12. **半透明 + 绊线**（`:1183-1186`）：
    ```java
    renderSectionLayer(RenderType.translucent(), ...);   // 含半透明排序
    renderSectionLayer(RenderType.tripwire(), ...);
    ```
13. **粒子、天气、手部**等后续阶段。

#### 3.4.3 renderSectionLayer（区块层绘制）

`LevelRenderer.renderSectionLayer`（`LevelRenderer.java:1280`）：

```java
renderType.setupRenderState();                          // 应用 RenderType 的所有 shard
// 半透明：必要时触发 section resortTransparency
ShaderInstance shaderinstance = RenderSystem.getShader();
shaderinstance.setDefaultUniforms(QUADS, frustumMatrix, projectionMatrix, window);
shaderinstance.apply();
Uniform uniform = shaderinstance.CHUNK_OFFSET;

for (RenderSection section : visibleSections) {         // 正向（不透明）或反向（半透明）
    if (!section.getCompiled().isEmpty(renderType)) {
        VertexBuffer vb = section.getBuffer(renderType);
        if (uniform != null) {
            uniform.set(originX - camX, originY - camY, originZ - camZ);  // CHUNK_OFFSET uniform
            uniform.upload();
        }
        vb.bind();
        vb.draw();                                       // 单次 drawElements
    }
}
shaderinstance.clear();
VertexBuffer.unbind();
renderType.clearRenderState();
```

关键点：每个 section 的 `VertexBuffer` 顶点是**局部坐标（0..16）**，通过 `CHUNK_OFFSET` uniform 平移到世界/相机空间，避免每帧重传顶点。

#### 3.4.4 SectionOcclusionGraph（遮挡剔除）

`LevelRenderer` 持有 `sectionOcclusionGraph`（`SectionOcclusionGraph`），基于 `CompiledSection.visibilitySet`（每个 section 6 个方向间的可见性，由 `VisGraph` 从不透明方块计算）做图遍历：从相机所在 section 出发，只标记"有视线通路"的邻居 section 为可见，大幅减少绘制与编译量。`addRecentlyCompiledSection`（`LevelRenderer.java:889`）在 section 编译完成时通知图更新。

### 3.5 GPU 上传

#### 3.5.1 两条上传路径

- **chunk 路径（STATIC VBO）**：`SectionCompiler` 在工作线程把顶点写入 `ByteBufferBuilder`（堆外）→ `BufferBuilder.build()` 产出 `MeshData` → 在渲染线程 `SectionRenderDispatcher.uploadSectionLayer`（`SectionRenderDispatcher.java:203`）调 `vertexBuffer.upload(meshData)` → `glBufferData(GL_ARRAY_BUFFER, ...)`。每个 `RenderSection` 的 5 个 `VertexBuffer` 是长期对象，数据驻留 GPU。
- **即时路径（DYNAMIC VBO）**：实体/BlockEntity/GUI/粒子通过 `MultiBufferSource.BufferSource.endBatch` → `RenderType.draw(meshData)` → `BufferUploader.drawWithShader`（`BufferUploader.java:24`）→ `VertexBuffer.upload` + `drawWithShader`，每帧重建。

#### 3.5.2 VertexBuffer.upload 细节

`VertexBuffer.upload(MeshData)`（`VertexBuffer.java:35`）：
1. `uploadVertexBuffer(drawState, vertexBuffer)`（`:112`）：若格式变化则 `clearBufferState` + `setupBufferState`（配置 VAO 属性指针），绑定 `GL_ARRAY_BUFFER` 并 `glBufferData` 传顶点字节。
2. `uploadIndexBuffer(drawState, indexBuffer)`（`:136`）：若有显式索引（半透明排序结果）则 `glBufferData(GL_ELEMENT_ARRAY_BUFFER, ...)`；否则取 `RenderSystem.getSequentialBuffer(mode)` 的共享顺序索引（QUADS 模式下自动生成 `0,1,2,3,4,5,...`）。

#### 3.5.3 draw call

`VertexBuffer.draw()`（`:161`）→ `RenderSystem.drawElements(mode.asGLMode, indexCount, indexType.asGLType)` 即 `glDrawElements`。区块用 `QUADS` → `GL_QUADS`，顺序索引由 `AutoStorageIndexBuffer` 维护。

`drawWithShader`（`:170`）额外在 draw 前后做 `shader.apply()`/`shader.clear()`，并 `setDefaultUniforms` 灌入矩阵。

## 4. 数据流总结

### 4.1 方块管线完整数据流

```
blockstate.json (variant/multipart)
   │  (ModelBakery 解析 BlockModelDefinition → Variant/MultiPart)
   ▼
block/*.json ──GSON+Deserializer──▶ BlockModel (UnbakedModel)
   │                                  ├ elements: List<BlockElement>
   │                                  ├ textureMap: Map<name, Material|#ref>
   │                                  ├ parent: BlockModel (继承链)
   │                                  └ transforms/overrides/guiLight
   ▼  BlockModel.bakeVanilla (spriteGetter 解析纹理 → TextureAtlasSprite)
for each BlockElement.face:
   FaceBakery.bakeQuad(from,to,face,sprite,dir,state,rot,shade)
      ├ setupShape (像素→1/16)
      ├ makeVertices: bakeVertex ×4 (applyElementRotation + applyModelRotation + fillVertex)
      ├ calculateFacing / recalculateWinding
      └ fillNormal
   ▼
BakedQuad (int[32] = 4顶点×8int, tint, dir, sprite, shade)   [格式=DefaultVertexFormat.BLOCK]
   ▼  SimpleBakedModel.Builder.addCulledFace/addUnculledFace
SimpleBakedModel (culledFaces: Map<Dir,List>, unculledFaces: List)  →  BlockModelShaper 缓存
   │
   │  ============ 切换到多线程 section 编译 ============
   ▼
SectionCompiler.compile (遍历 4096 方块)
   ├ ItemBlockRenderTypes.getRenderLayer(fluid) / model.getRenderTypes(state)
   │   → 选 RenderType ∈ {solid, cutoutMipped, cutout, translucent, tripwire}
   ├ getOrBeginLayer(rt) → BufferBuilder(byteBufferBuilder, QUADS, BLOCK)
   ├ blockRenderer.renderBatched → ModelBlockRenderer.tesselateBlock
   │     ├ getQuads(dir) + shouldRenderFace (剔除)
   │     ├ AmbientOcclusionFace.calculate (AO)
   │     └ putQuadData → consumer.putBulkData(pose, quad, brightness[], rgb, lightmap[], overlay)
   ├ liquidBlockRenderer.tesselate → buffer.addVertex().setColor().setUv().setLight().setNormal()
   └ buffer.build() → MeshData (vertexBuffer + DrawState)
   ▼
SectionRenderDispatcher.uploadSectionLayer (渲染线程)
   vertexBuffer.upload(meshData) → glBufferData(GL_ARRAY_BUFFER)
   ▼  RenderSection.buffers: Map<RenderType, VertexBuffer>  (驻留 GPU)
   │
   │  ============ 每帧绘制 ============
   ▼
LevelRenderer.renderSectionLayer(renderType, camX..)
   renderType.setupRenderState()  (shader/texture/blend/lightmap...)
   shader.setDefaultUniforms + apply
   for visibleSection:  shader.CHUNK_OFFSET.set(origin-cam);  vb.bind(); vb.draw()  → glDrawElements
   shader.clear(); renderType.clearRenderState()
```

```
   ▼
```

### 4.3 关键差异对比

| 维度 | 方块管线 | 实体管线 |
|------|---------|---------|
| 几何来源 | JSON 模型（数据驱动） | 代码 builder（编程式） |
| 烘焙时机 | 资源加载时 → `BakedQuad`；section 编译时写入缓冲 | 资源加载时 → `ModelPart.Cube`；每帧实时写入 |
| 顶点格式 | `DefaultVertexFormat.BLOCK`（32B，无 overlay） | `DefaultVertexFormat.NEW_ENTITY`（36B，含 overlay） |
| 缓冲驻留 | GPU `VertexBuffer`（STATIC，长期） | 即时 `ByteBufferBuilder`（每帧重建） |
| 光照 | AO + lightmap（编译时算入颜色/UV2） | lightmap + overlay（渲染时传 uniform/顶点） |
| 变换 | `CHUNK_OFFSET` uniform（section 级平移） | `PoseStack` 矩阵栈（逐骨骼） |
| 剔除 | 面剔除 + section 遮挡图 + 视锥 | 视锥 + `shouldRender` |
| 驱动类 | `SectionRenderDispatcher` + `SectionCompiler` | `LivingEntityRenderer` + `ModelPart` |
