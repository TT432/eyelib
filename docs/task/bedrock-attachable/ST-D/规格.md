# ST-D: 扩展 attachable 缓存到 EquipmentSlot

## 目标

把 `AttachableItemRenderSetup` 的缓存 key 从 `InteractionHand`（MAIN_HAND / OFF_HAND）改为 `EquipmentSlot`（MAINHAND / OFFHAND / HEAD / CHEST / LEGS / FEET），为 ST-E（盔甲 attachable）铺路。

## 前置条件

- ST-A/B/C 已完成（Resolver 条件评估、is_first_person、parent_setup）
- `AttachableItemRenderSetup.getOrPrepare(LivingEntity, InteractionHand, boolean)` 工作正常

## 后置条件

- 新增 `getOrPrepare(LivingEntity, EquipmentSlot, boolean)` 作为核心实现
- 旧 `getOrPrepare(LivingEntity, InteractionHand, boolean)` 改为 wrapper（映射 OFF_HAND→OFFHAND，其他→MAINHAND）
- 缓存类型 `Map<LivingEntity, EnumMap<EquipmentSlot, RenderData<ItemStack>>>`
- `context.item_slot` 按 EquipmentSlot 设置正确的 BE 字符串（main_hand / off_hand / slot.armor.head / slot.armor.chest / slot.armor.legs / slot.armor.feet）
- 现有所有调用点行为不变（手持路径）

## 不变量

1. 手持路径（InteractionHand wrapper）语义不变：`MAIN_HAND → EquipmentSlot.MAINHAND`、`OFF_HAND → EquipmentSlot.OFFHAND`
2. `context.item_slot` 字符串遵循 BE 规范和 `MolangBuiltInQuery.itemSlotToBoneName` 的映射约定（`main_hand`→rightitem、`off_hand`→leftitem、盔甲槽原样透传）
3. `invalidate(entity, slot)` 清除对应槽位缓存；`clearEntity(entity)` 清除整个实体的全部槽位
4. `tickForEntity` 仍只遍历手持（MAINHAND + OFFHAND）；盔甲槽 tick 由 ST-E 扩展

## 异常行为

- `getOrPrepare` 返回 null：无 attachable 时清理对应槽位的旧缓存（`invalidate(entity, slot)`）后返回 null
- `entity.getItemBySlot(slot)` 对非 LivingEntity 不会调用（方法签名限定 LivingEntity）

## 副作用

- `parent_setup` 仍在 holder scope 执行（行为不变，只是触发路径走 EquipmentSlot 核心方法）
- `context.is_first_person` 仍在 setup 时设置（缓存命中时更新）

## 设计决策

### D1: EquipmentSlot 而非自定义 SlotKey

用 JE 原生 `EquipmentSlot`（6 值枚举），不引入自定义类型。理由：
- JE 的 `LivingEntity.getItemBySlot(EquipmentSlot)` 是标准 API，直接对应 6 个装备槽
- `EnumMap<EquipmentSlot, ...>` 性能最优
- ST-E 的 ArmorLayer mixin 本就按 EquipmentSlot 工作

### D2: 旧 InteractionHand 重载保留为 wrapper

不删除旧签名，避免破坏所有现有调用点（ItemInHandRendererMixin、EntityRenderOrchestrator、AttachableSmoke、tickForEntity）。wrapper 内部 `hand == OFF_HAND ? OFFHAND : MAINHAND` 映射。

### D3: tickForEntity 保持手持范围

ST-D 不把 tickForEntity 扩展到盔甲槽。理由：
- ST-D 是结构重构，不改语义（职责单一）
- 盔甲 attachable 在 ST-E 才接入渲染层，tick 应与渲染同步扩展
- 提前 tick 盔甲槽会为未渲染的 attachable 浪费 CPU

### D4: item_slot 字符串映射表

集中在一个 `slotName(EquipmentSlot)` 私有方法：

| EquipmentSlot | item_slot 字符串 |
|---|---|
| MAINHAND | `main_hand` |
| OFFHAND | `off_hand` |
| HEAD | `slot.armor.head` |
| CHEST | `slot.armor.chest` |
| LEGS | `slot.armor.legs` |
| FEET | `slot.armor.feet` |

BE 规范定义，`itemSlotToBoneName` 对盔甲槽原样透传作为 bone name。

## 实施步骤

1. `AttachableItemRenderSetup.java`：
   - import `EquipmentSlot`
   - 缓存类型改 `EnumMap<EquipmentSlot, RenderData<ItemStack>>`
   - 新增 `getOrPrepare(LivingEntity, EquipmentSlot, boolean)` 核心方法（移入现有 setup 逻辑，用 `getItemBySlot` + `slotName`)
   - 旧 `getOrPrepare(LivingEntity, InteractionHand, boolean)` 改为 wrapper
   - 新增 `slotName(EquipmentSlot)` 工具方法
   - `invalidate` 签名改 `EquipmentSlot`
   - `tickForEntity` 遍历改为 `EquipmentSlot.MAINHAND, EquipmentSlot.OFFHAND`

2. 现有调用点无需改动（都走 InteractionHand wrapper）。
