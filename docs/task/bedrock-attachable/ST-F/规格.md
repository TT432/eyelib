# ST-F: enable_attachables 门控

## 前置条件

- ST-A/B/C/D/E 已完成
- `BrClientEntity.enable_attachables()` 字段已解析（boolean，默认 false）
- `RenderData.getClientEntityComponent().getClientEntity()` 可获取 holder 的 BrClientEntity

## 后置条件

- holder 的 BrClientEntity 存在且 `enable_attachables=false` 时，attachable 解析返回 null（不渲染）
- holder 无 BrClientEntity（vanilla 实体）时，默认允许 attachable
- holder 的 BrClientEntity `enable_attachables=true` 时，正常解析 attachable

## 不变量

- `enable_attachables` 检查发生在 item molang 条件评估之前（门控优先）
- vanilla 实体（无 eyelib client entity 定义）始终允许 attachable

## 异常行为

- 无新增异常

## 副作用

- 无新增副作用

## 设计决策

### 决策 1: 只实现 Client Entity 层面门控，不实现嵌套 attachable

**上下文**：BE 规范中 `enable_attachables` 有两个层面：
1. **Client Entity 层面**（ClientEntityDocumentationIntroduction.md L128）："This determines if the entity can equip attachables when this is set to true. This allows the entity to render armor and weapons."
2. **Attachable 层面**（attachable.md L45）："Whether to allow child attachables on this attachable."

**决策**：只实现 Client Entity 层面门控。

**理由**：
- Client Entity 层面门控简单、有用：eyelib 自定义实体可通过 `enable_attachables: false` 禁用 attachable 渲染
- 嵌套 attachable（attachable 上的 attachable）BE 规范不明确：attachable 不持有物品，无法通过 `AttachableResolver.resolve(holder, itemStack)` 解析子 attachable。BE vanilla 中几乎无此用法，文档仅一句 "Whether to allow child attachables on this attachable" 无实现细节
- 规划文档原述"在模型骨骼挂载点递归触发 attachable 解析"基于对嵌套语义的过度推测；实际 BE 规范不提供递归挂载的机制说明

### 决策 2: 检查位置在 AttachableResolver.resolve

**上下文**：门控检查可以放在 Resolver、getOrPrepare 或各 mixin 中。

**决策**：放在 `AttachableResolver.resolve(LivingEntity, ItemStack)` 中。

**理由**：
- 所有渲染路径（ItemInHandRendererMixin、HumanoidArmorLayerMixin、EntityRenderOrchestrator）都通过 resolve 判断是否有 attachable
- 在 resolve 中统一门控，避免各路径重复检查
- resolve 已有 holder 的 RenderData 访问（ST-A 重构），无需新增依赖

### 决策 3: hide_armor / hide_held_items 不在 ST-F 范围

**上下文**：BE 规范中 `hide_armor`（隐藏盔甲渲染）和 `hide_held_items`（Molang 条件隐藏手持物品）覆盖 `enable_attachables` 的渲染设置。

**决策**：不实现。

**理由**：BrClientEntity 未解析这两个字段。扩展 schema 解析超出 ST-F 范围（规划文档只提 `enable_attachables`）。
