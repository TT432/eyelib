# ST-B: 设置 context.is_first_person

## 前置条件
- ST-A 已完成：Resolver 能按 holder scope 评估 item molang 条件。

## 后置条件
- 第一人称路径（`ItemInHandRendererMixin.onRenderItem`）setup attachable 时 `context.is_first_person = 1.0`。
- 第三人称路径（`EntityRenderOrchestrator.renderItemInHand`）setup attachable 时 `context.is_first_person = 0.0`。
- `tickForEntity` 每次根据当前视角刷新 `context.is_first_person`（解决 F5 切换视角时缓存命中、setup 不重跑的问题）。

## 不变量
- `context.is_first_person` 的值在任意渲染帧内一致：setup 时写入 → tick 时（如有）覆盖 → renderAttachable 时读取到的总是当前帧的正确值。
- 只有本地玩家（`mc.player == entity`）在第一人称摄像机（`mc.options.getCameraType().isFirstPerson()`）下才为 `1.0`；其他实体恒为 `0.0`。

## 异常行为
- `Minecraft.getInstance()` 在非客户端线程返回的实例可能不一致；本方法只在客户端渲染线程被调用（mixin / TickStage 路径），无额外线程安全处理。
- `mc.player` 或 `mc.options` 为 null 时 `isLocalPlayerFirstPerson` 返回 false（null-safe）。

## 副作用
- **缓存命中时也会写入 scope**：`getOrPrepare` 即使走缓存命中分支，也会调 `updateFirstPerson` 覆盖 `context.is_first_person`。这是必要的——F5 切换视角不会触发 clientEntityChanged（attachable 没变），不覆盖的话 tick/animate 条件会读到旧值。

## 设计决策

### getOrPrepare 签名加 boolean isFirstPerson
`AttachableItemRenderSetup.getOrPrepare` 新增第三参数 `boolean isFirstPerson`，由调用方决定。理由：
- 第一人称路径（mixin）和第三人称路径（renderItemInHand）是不同的调用栈，调用方自然知道当前视角，传参最清晰。
- 避免 getOrPrepare 内部依赖 `Minecraft.getInstance()` 做"猜测"——一个 setup 方法不该关心调用方的视角语义。

### tickForEntity 内部判断视角
tick 路径（`TickStage.apply` → `tickForEntity`）不区分调用来源，统一用 `isLocalPlayerFirstPerson(entity)` 判断。理由：
- tick 跑在所有 LivingEntity 上（不只是本地玩家），对非本地玩家恒返回 false 是正确的。
- 对本地玩家，tick 时视角可能与上一帧不同（F5），必须刷新。

### renderAttachable 不加参数
renderAttachable 读取 scope 中已有的 `context.is_first_person`（由 setup/tick 维护），不接收参数。理由：render 不应该改变上下文，只读。

### 工具演化：IgnoreLineNumberViolationMatcher（衍生工作）
ST-B 期间第二次遇到 ArchUnit freeze baseline 因行号偏移误报 new violation（第一次在 ST-A）。根本问题是 ArchUnit 默认的 `FuzzyViolationLineMatcher` 把行号纳入匹配字符串。

**方案**：实现自定义 `ViolationLineMatcher`，正则剥离 ` in (File.java:N)` 中的 `:N` 后比较。通过 `archunit.freeze.lineMatcher` 系统属性启用。

**取舍**：同一方法内多处调用同一目标类会被合并为一条 baseline 条目（因为去掉行号后描述相同）。这是可接受的——架构规则关心的是"依赖关系是否存在"，而非"出现在哪一行"。真正的违规（新的依赖关系）仍会被检测。

**后续 ST 受益**：ST-C/D/E 改动源码行号不再需要重新冻结 baseline。

## 实施步骤
1. `AttachableItemRenderSetup.java`：
   - `getOrPrepare` 签名加 `boolean isFirstPerson`
   - setup 分支：`scope.set("context.is_first_person", isFirstPerson ? 1F : 0F)`
   - 缓存命中分支：调 `updateFirstPerson(rd, isFirstPerson)`
   - 新增 `isLocalPlayerFirstPerson(Entity)` 静态方法
   - `tickForEntity` 内部调 `getOrPrepare(entity, hand, isLocalPlayerFirstPerson(entity))`
2. `ItemInHandRendererMixin.java:42`：调 `getOrPrepare(entity, hand, true)`
3. `EntityRenderOrchestrator.java:284`：调 `getOrPrepare(le, hand, false)`
4. `AttachableSmoke.java:51`：调 `getOrPrepare(player, MAIN_HAND, true)`
5. 新增 `IgnoreLineNumberViolationMatcher`（test 源码）
6. `build.gradle` test 任务加 `archunit.freeze.lineMatcher` 系统属性
7. 删除旧 baseline，重新冻结
