# ST-A: Resolver 评估 item 条件 molang

## 目标

`AttachableResolver` 在解析 attachable 时，按 holder 作用域评估 `description.item` 字段 value 的 molang 条件表达式，truthy 才匹配。对齐 MC BE 规范（attachable.md：item value 是 molang，如 `"query.is_owner_identifier_any('minecraft:player')"`）。

## 前置条件

- `AttachableManager.INSTANCE` 含若干 `BrClientEntity`，每个 attachable 的 `item()` 是 `Map<String, String>`（itemId → molang 条件源码）。
- 调用方传入的 `LivingEntity holder` 已通过 `RenderData.getComponent(holder)` 拿到 RenderData（可能未 init）。

## 后置条件

- `AttachableResolver.resolve(LivingEntity, ItemStack)` 评估 item molang 条件，仅 truthy 才返回 attachable。
- 新增 `resolveByItemId(String, MolangScope)` 公开 API，评估条件。
- 保留 `resolveByItemId(String)` 旧 API（存在性检查），行为不变。

## 不变量

1. `resolveByItemId(String)` 语义不变：`attachable.item().containsKey(itemId)` 即匹配。
2. `resolveByItemId(String, MolangScope)` 对常量 `"1.0"` 在任意非 null scope 下都匹配。
3. `resolveByItemId(String, MolangScope)` 对 `"0.0"` 在任意非 null scope 下都不匹配。
4. `resolveByItemId(String, MolangScope)` 引用 `variable.xxx` / `query.xxx` 的条件按 scope 中的 holder 上下文评估。
5. 遍历顺序遵循 `AttachableManager.INSTANCE.all().values()` 的迭代顺序，返回第一个 truthy 匹配。

## 异常行为

- molang 评估抛异常时，`MolangValue.getObject` 已 try-catch 返回 `MolangNull.INSTANCE`，`evalAsBool` 返回 false → 该 attachable 不匹配（fail-fold，不中断遍历）。
- `resolveByItemId(String, MolangScope)` 接收 null scope：抛 NPE（包级 `@NullMarked` 约定，scope 参数非空）。

## 副作用

- `resolve(LivingEntity, ItemStack)` 在 holder 的 RenderData 未 init（`getScope() == null`）时调 `rd.ensureOwner(holder)` 触发 lazy-init：创建 scope、put host context（RenderData.class + owner.getClass()）、set `variable.scale=1`。这是必要副作用，保证 scope 可用且 holder 上下文可被 `query.is_owner_identifier_any` 等访问。

## 设计决策

### scope 来源：holder 的 RenderData scope

- 选 `RenderData.getComponent(holder).getScope()`，不创建临时 scope。
- 理由：item molang 可能引用 holder variable（如 `variable.xxx`）或 query（如 `query.is_owner_identifier_any`），用 holder 的真实 scope 最正确；holder 在渲染流程中本来就会 init RenderData，lazy-init 不是新副作用（只是提前触发）。
- 调用方语义：`resolve` 接收 holder 是为了拿这个 scope，不是单纯查 item。

### 保留旧 `resolveByItemId(String)` API

- 旧 API 保持存在性检查语义，不评估条件。
- 理由：
  1. 向后兼容现有测试（`AttachableResolverTest` 6 处调用，其中 `resolveByItemIdHandlesMultipleItemsInOneAttachable` 的 `"demo:axe"` 条件是 `"query.is_sneaking"`，存在性检查能匹配）。
  2. 用于不需要条件过滤的场景（如纯 itemId 索引查询）。
- 新增 `resolveByItemId(String, MolangScope)` 是条件评估 API，是 `resolve(LivingEntity, ItemStack)` 的底层。

### item value 构造 MolangValue

- 每次 `resolveByItemId(String, MolangScope)` 都 `new MolangValue(conditionStr)`。
- `MolangValue(String)` 走 `MolangCompileCache`，重复构造无开销（编译结果缓存）。
- 不在 `BrClientEntity` 解析时预编译 item value：item value 是 molang 源码字符串，预编译会改变 `BrClientEntity` 的数据模型（record 字段类型变化），影响范围过大。运行时构造 + 缓存足够。

## 实施步骤

1. 修改 `AttachableResolver.java`：
   - import `MolangScope`、`MolangValue`、`RenderData`。
   - `resolve(LivingEntity, ItemStack)`：拿 itemId 后，从 `RenderData.getComponent(holder)` 取 rd，若 `rd.getScope() == null` 调 `rd.ensureOwner(holder)`，然后 `resolveByItemId(itemId, rd.getScope())`。
   - 新增 `resolveByItemId(String itemId, MolangScope scope)`：遍历 attachable，`item().get(itemId)` 非 null 时 `new MolangValue(value).evalAsBool(scope)` truthy 即返回。
   - 保留 `resolveByItemId(String itemId)` 不变。
2. 新增单测覆盖条件评估。
