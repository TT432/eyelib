# 任务：在 eyelib 完整对齐 MC BE Attachable 规范

## 任务目标

将 eyelib 现有的 attachable 实现（仅覆盖手持物品场景）扩展到完整对齐 MC 基岩版 attachable 规范。
覆盖：手持物品（第一/第三人称）、盔甲槽穿戴、Molang 上下文、scripts 全字段、嵌套 enable_attachables。

## 差距清单（与 BE 规范对比）

### G1. `AttachableResolver` 未评估 item 条件 molang
- **现状**：`resolveByItemId(itemId)` 仅检查 `attachable.item().containsKey(itemId)`。
- **规范**：`item` 字段支持 `{"item_id": "molang_condition"}` 形式（见 wrench 示例：`{"demo:wrench":"query.is_owner_identifier_any('minecraft:player')"}`）。Molang 在 **holder 实体作用域**评估，truthy 才匹配。
- **影响**：当前所有 attachable 都会被任意 holder 持有，无法按实体类型/状态过滤。

### G2. `context.is_first_person` 未设置
- **现状**：`AttachableItemRenderSetup.getOrPrepare` 只设 `context.item_slot`，未设 `context.is_first_person`。
- **规范**：BE attachable 通过 `context.is_first_person`（1.0/0.0）区分第一/第三人称，常用于 `scripts.animate` 条件（如 wrench 示例：`[{"hold_first_person":"context.is_first_person == 1.0"},...]`）。
- **影响**：第一/第三人称动画切换完全失效。

### G3. `scripts.parent_setup` 未调用
- **现状**：`BrClientEntityScripts.parent_setup` 已建模（codec 可解析），但 `EntityRenderOrchestrator.setupClientEntity`（line 446-454）在 `clientEntityChanged` 时只调 `initialize` + `pre_animation`，**未调 `parent_setup`**。
- **规范**：parent_setup 是 Molang，在 **holder 实体的 MolangScope**（不是 attachable 自己的 scope）执行，用于初始化父变量（典型：`variable.chest_layer_visible = 0.0;`）。
- **影响**：依赖父变量初始化的 attachable（特别是盔甲 attachable 隐藏 vanilla 胸甲层）行为不正确。

### G4. 盔甲槽 attachable 完全缺失
- **现状**：无 ArmorLayer/HumanoidArmorLayer mixin；`AttachableResolver` 无按 `EquipmentSlot` 解析；`AttachableItemRenderSetup` 缓存 key 只支持 `InteractionHand`。
- **规范**：BE attachable 的 `context.item_slot` 取值包括 `slot.armor.head`、`slot.armor.chest`、`slot.armor.legs`、`slot.armor.feet`、`slot.armor.offhand`（elytra 走 chest 槽）。Attachable 通过 `binding`（Molang）绑定到 holder 模型的对应骨骼（如 `armorR`/`armorL`/`helmet` 等）。
- **影响**：所有盔甲 attachable（chestplate/helmet/leggings/boots/elytra 的自定义模型）无法工作。

### G5. `enable_attachables` 嵌套未实现
- **现状**：`BrClientEntity` 已解析 `enable_attachables` 字段，但渲染路径未消费。
- **规范**：当 client entity（包括 attachable 自身）的 `enable_attachables=true` 时，其骨骼上挂载的物品也应触发 attachable 解析与渲染（递归）。
- **影响**：嵌套场景失效（少见，但规范要求）。

### G6. 双重 tick 风险（待验证）
- **现状**：
  - `TickStage.apply`（EntityRenderOrchestrator.java:170）每帧调 `AttachableItemRenderSetup.tickForEntity`。
  - `ItemInHandRendererMixin.onRenderItem`（line 47-69）在 `ac.tickedInfos == null` 时**再次 tick**。
- **风险**：第一人称路径可能在 TickStage 之外触发额外 tick（动画速度翻倍 / 粒子重复）。
- **验证目标**：确认第一人称 attachable 的 tick 来源唯一。

### G7. `render_controllers` 条件形式（待验证）
- **现状**：`EntityRenderOrchestrator.setupClientEntity`（line 463-469）已实现条件评估（`renderControllerConditions.get(rc).evalAsBool(scope)`）。
- **验证目标**：确认 attachable 路径也走同一 `setupClientEntity`，条件正确生效（目前看代码路径是共享的，应已工作）。

## 子任务分解

子任务设计原则：**可验证 + 互相孤立 + 按依赖排序**。

| ID | 子任务 | 依赖 | 主要产出 | 验证闸门 |
|---|---|---|---|---|
| **ST-A** | Resolver 评估 item 条件 molang | 无 | `AttachableResolver.resolve` 重构：按 holder 作用域评估 item molang | 单测 |
| **ST-B** | 设置 `context.is_first_person` | 无 | `AttachableItemRenderSetup` 在第一/第三人称路径分别写入 1.0/0.0 | 单测 + clientsmoke |
| **ST-C** | 实现 `scripts.parent_setup` 调用 | 无 | `setupClientEntity` 路径在 holder scope 执行 parent_setup | 单测 |
| **ST-D** | 扩展 attachable 缓存到 EquipmentSlot | ST-A | `AttachableItemRenderSetup` 缓存 key 从 `(LivingEntity, InteractionHand)` 改为 `(LivingEntity, EquipmentSlot)`；新增 `getOrPrepare(le, slot)` 重载 | 单测 |
| **ST-E** | ArmorLayer mixin 实现盔甲 attachable 渲染 | ST-D, ST-B, ST-C | mixin 注入盔甲渲染路径，按 EquipmentSlot 解析 attachable 并渲染（cancel vanilla 盔甲模型） | clientsmoke |
| **ST-F** | enable_attachables 嵌套支持 | ST-E | 在模型 visitor 骨骼挂载点递归触发 attachable 解析 | 单测 + clientsmoke |
| **ST-G** | clientsmoke 全场景覆盖 + 文档同步 | 全部 | 更新 `AttachableSmoke`，新增盔甲/第一人称区分用例；同步 docs | clientsmoke 全绿 |

### 依赖图

```
ST-A (Resolver molang) ─┐
ST-B (is_first_person) ─┼─→ ST-D (EquipmentSlot 缓存) ─→ ST-E (ArmorLayer mixin) ─→ ST-F (嵌套) ─→ ST-G (验证+文档)
ST-C (parent_setup) ────┘                                            │

```

ST-A/B/C 可并行（互相独立）。ST-D 汇聚三者。ST-E 是最大工作量。

## 关键决策点（待执行时确认）

1. **第一人称路径确认**：`ItemInHandRendererMixin` 注入的 `ItemInHandRenderer.renderItem(LivingEntity, ItemStack, ItemDisplayContext, boolean, PoseStack, MultiBufferSource, int)` 是否 100% 是第一人称专用？若是，则在该路径设 `is_first_person=1.0`，第三人称路径（`EntityRenderOrchestrator.renderItemInHand`）设 `0.0`。需对照 JE 源码确认。

2. **parent_setup 的 scope 选择**：BE 规范要求"在父实体作用域执行"。eyelib 中 holder 实体的 MolangScope 在 `RenderData.getComponent(holder).getScope()`。attachable setup 时需读 holder 的 RenderData scope 执行 parent_setup，再切回 attachable 自己的 scope 继续。需确认 holder 一定有 RenderData（玩家/生物应有；非 LivingEntity 的 holder 边界情况）。

3. **盔甲 attachable 的 vanilla cancel 策略**：JE 的 `HumanoidArmorLayer.renderModel` 会渲染 vanilla 盔甲模型。mixin 应在 HEAD cancel 并渲染 attachable，还是修改其内部让 vanilla 渲染跳过？需看 `HumanoidArmorLayer` 1.20.1 源码结构决定（`RenderHumanoidArmorLayer` / `HumanoidArmorLayer` 类名 + `renderArmorPiece` 方法签名）。

4. **elytra 特殊处理**：vanilla 有单独的 `CapeLayer`/`WingsLayer`（elytra 通过 `ElytraLayer`）。BE 中 elytra attachable 走 `slot.armor.chest`。是否需要在 `ElytraLayer` 也加 mixin，还是统一在 `HumanoidArmorLayer` 的 CHEST 分支处理？需调查。

5. **第一人称 attachable 的骨骼绑定**：第一人称路径没有 holder 的完整模型（只渲染手）。BE 第一人称 attachable 的 `binding`（如 `q.item_slot_to_bone_name(context.item_slot)`）在 eyelib 第一人称路径如何提供骨骼？现状 `ItemInHandRendererMixin` 直接在 vanilla poseStack 上渲染 attachable，未做骨骼绑定。这可能是现有手持实现的一个隐性缺陷（手持 attachable 可能不需要复杂骨骼绑定，但需验证）。

## 风险与回退

- **ArmorLayer mixin 跨版本差异**：1.20.1（Forge）和 1.21.1（NeoForge）的盔甲渲染类名/方法签名可能不同。用 Stonecutter `//?` 注释切分版本特定代码。
- **parent_setup scope 污染**：在 holder scope 执行 parent_setup 后，需确保不残留临时变量影响 holder 自身渲染。可能需要 scope 保存/恢复。
- **clientsmoke 盔甲场景构造**：clientsmoke 在 GUI/单线程环境构造穿戴盔甲的实体可能有限制，需读现有 `AttachableSmoke` 模式。
