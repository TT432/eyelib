# ST-C: 实现 scripts.parent_setup 调用

## 前置条件
- ST-A 已完成：Resolver 能评估 item molang 条件。
- ST-B 已完成：context.is_first_person 已正确设置。

## 后置条件
- attachable 首次绑定到 holder 时，其 `scripts.parent_setup` 在 **holder（父实体）的 scope** 上执行。
- holder 的 Molang 变量（如 `variable.chest_layer_visible`）被 parent_setup 初始化。

## 不变量
- parent_setup 只在 attachable 变化时执行一次（`getOrPrepare` 的非缓存路径），不会每帧重复执行。
- parent_setup 在 holder 的 RenderData scope 上执行，**不在** attachable 的 RenderData scope 上执行。这是 BE 规范的关键语义——parent_setup 用于修改父实体的渲染状态。

## 异常行为
- holder 的 RenderData scope 未初始化时，`ensureOwner(entity)` 会 lazy-init。这是幂等操作，已在 ST-A 中用于 Resolver。
- parent_setup 表达式抛异常时由 Molang 引擎处理（不影响渲染流程）。

## 副作用
- holder 的 RenderData scope 被写入 parent_setup 设置的变量。这些变量会影响 holder 自身的渲染（如隐藏胸部图层）。

## 设计决策

### 放在 getOrPrepare 而非 setupClientEntity
`EntityRenderOrchestrator.setupClientEntity` 是共享方法，只接收 attachable 的 `RenderData cap`，不知道 holder。parent_setup 需要 holder 的 scope，所以放在 `AttachableItemRenderSetup.getOrPrepare` 中（它持有 holder 引用）。

执行位置：`setupClientEntity(attachable, rd)` 之后。理由：
- setupClientEntity 内部 `consumeChanged()` 检测 clientEntityChanged，执行 attachable scope 的 initialize/pre_animation。
- parent_setup 是 holder scope 的操作，逻辑上独立，放在 setupClientEntity 之后。
- 此路径只在非缓存分支（新 attachable 绑定）走到，保证 parent_setup 只执行一次。

### holder scope 来源
`RenderData.getComponent(entity)` → `ensureOwner(entity)` → `requireScope()`。

与 ST-A 的 Resolver 使用相同的 scope 获取模式。holder 的 RenderData 是 entity 级别的，与 attachable 的 item 级别 RenderData 是不同对象。

### 多 attachable 场景
main_hand 和 off_hand 的 attachable 可能都有 parent_setup。它们在各自的 `getOrPrepare` 中执行，后执行的覆盖先执行的 holder 变量。这是 BE 规范的合理行为——多个 attachable 的 parent_setup 都在 holder scope 上执行。

## 实施步骤
1. `AttachableItemRenderSetup.java`：在 `setupClientEntity(attachable, rd)` 之后，获取 holder RenderData，执行 `s.parent_setup().eval(holderRd.requireScope())`。
