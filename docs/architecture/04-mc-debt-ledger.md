# Functional Module Debt Ledger

## Purpose
- Record debt caused by organizing Minecraft-related code around `mc/api` and `mc/impl` instead of feature ownership.
- Reframe the refactor target: this is a Minecraft mod, so Minecraft/Forge usage is not a boundary violation by itself.
- Use functional modules as the primary ownership rule: particle code belongs to particle, material code belongs to material, importer code belongs to importer, attachment code belongs to attachment, and so on.

## Direction
- Do not preserve `mc/api` and `mc/impl` as architectural constraints.
- Do not move classes merely to isolate Minecraft/Forge imports.
- Prefer feature cohesion over platform isolation.
- If a class is only used by one feature module, move it into that module even if it imports Minecraft or Forge types.
- Keep root as the assembled mod surface only when code genuinely coordinates multiple feature modules or is required by the root mod entrypoint.
- Extract a new submodule only when a functional area has a stable responsibility that is not owned by an existing module.
- Split a class when it mixes several feature responsibilities, not because it imports Minecraft runtime types.

## Functional Ownership Rule

| Code Kind | Preferred Owner |
|---|---|
| Particle definitions, spawn requests, particle packets, particle render/runtime helpers | `:eyelib-particle` unless multiple non-particle features own the behavior |
| Material definitions, render state, shader/material runtime | `:eyelib-material` |
| Bedrock/imported schemas, addon discovery, parsed model/entity/animation/particle source data | `:eyelib-importer` |
| Attachment data contracts, attachment storage, attachment codecs, attachment sync contracts | `:eyelib-attachment` or a follow-up attachment-owned runtime package |
| Molang compiler/runtime/mapping behavior | `:eyelib-molang` |
| Root mod startup that wires independent modules together | root runtime |
| Compatibility facades needed only for old root callers | root, temporary and explicitly marked |

## Current Debt Themes

| Theme | Why It Blocks Functional Modules | Target Shape |
|---|---|---|
| `mc/api` bridge contracts | The package describes platform layering, not the feature that owns the contract. | Move each contract to the feature that owns it, or delete it if it only hides a same-module call. |
| `mc/impl` as a bucket | Feature code is hidden under an implementation bucket because it touches Minecraft types. This makes ownership unclear. | Move classes to their functional module/package even when they use Minecraft/Forge APIs. |
| Root packet DTO ownership | Feature-specific packets live in root `mc/impl/network/packet`, so particle and attachment cannot own their own protocol surface. | Move feature-specific packet contracts/codecs to the owning feature module when practical. |
| Attachment split drift | `:eyelib-attachment` exists, but root still owns many attachment contracts and sync pieces. | Consolidate attachment-owned data, storage, codecs, and sync contracts around the attachment module. |
| Particle compatibility leftovers | Root still holds particle spawn, legacy runtime, lookup, manager, and packet adapters. | Move particle-only runtime and protocol code into `:eyelib-particle`; keep only temporary root compatibility facades if still needed. |
| Importer plain-library split | Importer now has an explicit current identity as the `eyelibimporter` importer/schema Forge functional module, but that means it is not yet a separate plain importer library. | Keep Forge bootstrap isolated and split a plain importer core later only if a product requirement needs non-Forge consumption. |
| Documentation mismatch | Existing docs still describe a runtime isolation model. | Rewrite docs around functional ownership and remove `mc/api` / `mc/impl` as boundary rules. |

## Suggested First Cleanup Slices

1. Rewrite docs to remove `mc/api` and `mc/impl` as boundary rules and replace them with functional ownership rules.
2. Register `:eyelib-attachment` in `MODULES.md` before moving attachment code.
3. Move `ManagerEventPublisher` and its Forge publisher into client manager ownership as a small proof-of-direction change.
4. Classify particle root leftovers into keep-as-facade, move-to-particle, or delete-after-callers-migrate.
5. Classify attachment packet/data/event classes into attachment-owned contracts versus feature runtime observers.

## Non-Goals For This Ledger
- This document does not say Minecraft imports are bad.
- This document does not preserve runtime isolation as a goal.
- This document does not require every feature to become a submodule immediately.
- This document does not decide whether importer or attachment should be plain JVM libraries or Minecraft functional modules; it records that those decisions are currently ambiguous.
