# Bedrock Add-On 都有哪些属性

## 1. 范围说明

这里的“属性”优先指 Bedrock Add-On / Pack 在**容器层**和 **manifest 层**公开表达出来的字段，不试图在第一版里穷尽所有业务 JSON 文件的每一个字段。

本页重点覆盖：

1. addon 或 pack 的容器属性。
2. `manifest.json` 的顶层属性。
3. `header`、`modules`、`dependencies`、`metadata`、`capabilities`、`settings` 这些规范对象的常见属性。

## 2. 容器与分发层有哪些“属性”

从官方文档和社区文档看，Bedrock Add-On 在被导入、分发、落盘时，至少有这些属性维度：

| 属性维度 | 含义 | 常见取值 / 形式 | 主要来源 |
|---|---|---|---|
| 容器类型 | 内容是如何打包与导入的 | 文件夹、`.mcpack`、`.mcaddon` | O1, C1 |
| 资产类型 | 单包还是复合包 | 单个 pack、多个 pack 组合 | C1 |
| 包类型 | pack 本身属于什么模块 | `resources`、`data`、`script`、`world_template` 等 | O5 |
| 安装位置 | Minecraft 导入后放到哪里 | `resource_packs`、`behavior_packs`、开发目录 | O1, C2 |
| 依赖关系 | 一个 pack 是否依赖另一个 pack 或脚本模块 | `dependencies` 中的 UUID / module_name / script package | O5, R3, R4, R5 |

## 3. `manifest.json` 顶层有哪些属性

根据 Microsoft Learn 的 `manifest.json` 参考，`manifest.json` 顶层常见属性至少包括：

| 顶层属性 | 是否常见 / 核心 | 作用 | 主要来源 |
|---|---|---|---|
| `format_version` | 核心 | 说明 manifest 自身语法版本 | O5, O6 |
| `header` | 核心 | pack 身份、版本、显示信息等 | O5, O6 |
| `modules` | 核心 | 描述 pack 内部模块及其类型 | O5, O6 |
| `dependencies` | 常见 | 声明对其他 pack 或模块的依赖 | O5, O6 |
| `capabilities` | 可选 | 开启某些能力，如 PBR、editorExtension 等 | O5, R5, R6 |
| `metadata` | 可选但常见 | 作者、许可证、URL、生成工具、产品类型等 | O5, R3, R4, R5, R6 |
| `settings` | 较新 / 特定版本 | manifest v3 相关的设置界面能力 | O5 |

## 4. `header` 有哪些属性

`header` 是最基础的一组属性，决定 pack 的身份和展示方式。

| 属性 | 含义 | 备注 | 主要来源 |
|---|---|---|---|
| `name` | 包名 | 可直接写文本，也可能配合语言文件 | O5, C2 |
| `description` | 包描述 | 常在游戏内显示 | O5, C2 |
| `uuid` | 包唯一标识 | 同一个 pack 版本族的核心身份字段 | O5, C3 |
| `version` | 包版本 | 文档与社区资料里都非常关键 | O5, O6, C2 |
| `min_engine_version` | 最低引擎版本 | 决定兼容范围 | O5, O6, C2 |
| `pack_scope` | 资源包作用域 | 常见于 RP，如 `world` | O5, R4, C2 |
| `base_game_version` | 基础游戏版本 | 官方文档列出 | O5 |
| `allow_random_seed` | 是否允许随机种子设置 | 官方文档列出 | O5 |
| `lock_template_options` | 是否锁定模板选项 | 主要和模板相关 | O5 |

## 5. `modules` 有哪些属性

`modules` 用来描述一个 manifest 里承载了哪些模块。第一版文档只整理结构，不展开到脚本 API 细节。

| 属性 | 含义 | 常见取值 / 形式 | 主要来源 |
|---|---|---|---|
| `description` | 模块描述 | 文本 | O5 |
| `type` | 模块类型 | `resources`、`data`、`script`、`world_template` | O5, O6 |
| `uuid` | 模块 UUID | 唯一标识模块 | O5, O6 |
| `version` | 模块版本 | 与 header 版本相关但不是同一字段 | O5, O6 |
| `language` | 脚本语言 | 常见于 script module | O5, R3, R4, R5, R6 |
| `entry` | 脚本入口文件 | 常见于 script module | R3, R4, R5, R6 |

## 6. `dependencies` 有哪些属性

`dependencies` 描述 pack 对其他 pack 或模块的依赖关系。

| 属性 | 含义 | 备注 | 主要来源 |
|---|---|---|---|
| `uuid` | 依赖 pack 的 UUID | 最常见的 BP/RP 互相依赖方式 | O5, O6, R3, R4 |
| `version` | 依赖版本 | 可能是数组，也可能是其他受支持形式 | O5, O6 |
| `module_name` | 依赖模块名 | 常见于脚本模块依赖 | O5 |

从真实样例看，`dependencies` 至少有三类用途：

1. BP 依赖 RP。
2. RP 依赖 BP。
3. 脚本型 addon 依赖 `@minecraft/server`、`@minecraft/server-ui` 等模块。

## 7. `metadata` 有哪些属性

`metadata` 不是所有 pack 都写，但在较完整的真实样例里很常见。

| 属性 | 含义 | 主要来源 |
|---|---|---|
| `authors` | 作者列表 | O5, R5, R6, C2 |
| `license` | 许可证 | O5, R5, R6 |
| `generated_with` | 使用哪些工具生成 | O5, R3, R4 |
| `product_type` | 产品类型，如 `addon` | O5, C2 |
| `url` | 项目主页或仓库地址 | O5, R5 |

## 8. `capabilities` 有哪些属性

`capabilities` 用来声明某些额外能力。

官方文档列出的能力至少包括：

- `chemistry`
- `editorExtension`
- `experimental_custom_ui`
- `raytraced`

真实样例和社区仓库里还能看到资源包使用：

- `pbr`

这里要注意：

1. 不是每个 pack 都会写 `capabilities`。
2. 社区样例中出现的能力，不一定都能当成“广义官方稳定规范”；需要结合官方文档和目标版本判断。

## 9. `settings` 有哪些属性

`settings` 是 manifest 新版本下的一个扩展方向，主要和包的设置界面有关。官方文档指出它是一个有顺序的设置项列表，并有不同类型的设置项，如：

- label
- toggle
- slider

这一组属性说明 manifest 不再只是“识别 pack”，也开始承载“面向用户的配置界面定义”。

## 10. 真实样例里出现了哪些额外属性组合

从真实样例可以看出，除了最小教程包，实际 addon 往往还会引入这些组合：

| 场景 | 常见属性组合 | 主要样例 |
|---|---|---|
| 最小 BP | `format_version + header + modules(data)` | R1 |
| 最小 RP | `format_version + header + modules(resources)` | R2 |
| 成对 BP/RP | `dependencies` 把 BP 与 RP 绑在一起 | R3, R4 |
| 脚本型 addon | `modules` 中新增 `script`，并配合脚本依赖 | R3, R4, R5, R6 |
| 富元数据 addon | `metadata.authors/license/url/product_type` | R5, R6 |
| 高级资源包 | `capabilities: ["pbr"]` | R5, R6 |

## 11. 第一版结论

如果只看“属性”，Bedrock Add-On 绝对不只是一个简单的 `manifest.json` 文件，而是至少包含四层属性体系：

1. **容器属性**：文件夹、`.mcpack`、`.mcaddon`。
2. **pack 属性**：RP / BP / script / template 等模块与依赖关系。
3. **manifest 属性**：`header`、`modules`、`dependencies`、`metadata`、`capabilities`、`settings`。
4. **资源文件属性**：各目录下 JSON、图像、语言文件、声音定义文件等各自的结构化字段。

本页只完成到第 1 步，把 pack 级和 manifest 级属性先立住。更细的资源文件属性，应在后续专题页继续展开。
