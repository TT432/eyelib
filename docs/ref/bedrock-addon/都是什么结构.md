# Bedrock Add-On 都是什么结构

## 1. 使用范围

本页把 Bedrock Add-On 的外层组织方式整理成**结构参考矩阵**。

这里的“结构”只讨论：

- 导入容器
- pack 组合
- 目录族
- 文件族

不讨论每个业务 JSON 文件内部的字段树。也就是说：

- `manifest.json` 的字段 → 看 `都有哪些属性.md`
- `.mcaddon` / `.mcpack` / `resource_pack/` / `behavior_pack/` 这些层级 → 看本页

## 2. 读表规则

- **层级**：这个结构属于 Add-On 容器层、pack 层，还是 pack 内容层。
- **类型**：它是容器、pack、目录、文件，还是样例中出现的附属路径。
- **规范性**：
  - `stable`：官方稳定文档或官方工作流明确支持。
  - `preview`：官方 preview / schema 语义。
  - `sample-only`：真实样例中出现，但不应直接当成稳定标准入口。
- **主要参考**：见 `来源清单.md`。

## 3. Add-On 总层级矩阵

| 名称 | 层级 | 类型 | 作用 | 常见范围 | 规范性 | 主要参考 | 备注 |
|---|---|---|---|---|---|---|---|
| Add-On | 最外层 | 逻辑集合 | 组合一个或多个可导入 pack | 单包或多包 | stable | O1, O2, C1 | 不是单一 JSON 格式，而是分层系统 |
| 容器层 | Add-On 层 | 组织层 | 决定内容如何被打包与导入 | 文件夹 / `.mcpack` / `.mcaddon` | stable | O1, C1 | 不是 manifest 字段 |
| Pack 层 | Add-On 层 | 组织层 | 把内容分成 RP / BP / script module 等 | 单 pack 或多 pack | stable | O2, O3, O4 | “一个 addon 有几个 pack”属于这一层 |
| 内容层 | Pack 层 | 组织层 | 放置 `manifest.json`、`pack_icon.png`、目录族、具体资源文件 | RP/BP 内容树 | stable | O2, O3, O4, R7, R8 | 本页核心关注对象 |

## 4. 容器结构矩阵

| 名称 | 层级 | 类型 | 作用 | 常见范围 | 规范性 | 主要参考 | 备注 |
|---|---|---|---|---|---|---|---|
| 文件夹 | 容器层 | 容器 | 开发态最常见形态 | 开发目录、导入目录 | stable | O1, C2 | 文件夹是最基础的工作流形态 |

| `.mcaddon` | 容器层 | 复合导入容器 | 一次导入多个资产归档或顶层资产目录 | 多包分发 | community clarified | C1 | 社区与工具链普遍按复合 zip 容器理解；当前官方稳定文档没有像 `.mcpack` 那样单独给出等价定义 |

| `resource_packs/` | 工作流层 | 导入目录 | 导入后资源包落盘位置 | RP 导入态 | stable | O1 | 不属于 manifest 结构 |

## 5. Pack 组合矩阵

| 名称 | 层级 | 类型 | 作用 | 常见范围 | 规范性 | 主要参考 | 备注 |
|---|---|---|---|---|---|---|---|
| Behavior Pack (BP) | Pack 层 | pack | 承载玩法/逻辑/规则定义 | `behavior_pack/` 或 BP 目录 | stable | O2, O4, R8 | 经常和 RP 配套，但不是绝对必须 |
| Resource Pack (RP) | Pack 层 | pack | 承载视觉、声音、文本等客户端资源 | `resource_pack/` 或 RP 目录 | stable | O2, O3, R7 | 经常和 BP 配套，但不是绝对必须 |
| script module | Pack 层 / module 层 | module | 为 pack 提供脚本入口与脚本依赖 | `modules[].type = script` | stable | O2, O5-MODULE, R3, R4 | 它是 module 语义，不等于目录 `scripts/` |
| BP + RP 成对组合 | Add-On 层 | 常见组合 | 构成常见“完整 addon”体验 | 标准成对 addon | stable/common practice | O2, R3, R4 | 应写成“常见结构”，不能写成“强制规则” |
| 单 BP 样例 | Add-On 层 | 常见组合 | 演示纯行为层能力 | `behavior_pack_sample` | stable sample | R1 | 证明单包也是合法结构 |
| 单 RP 样例 | Add-On 层 | 常见组合 | 演示纯资源层能力 | `resource_pack_sample` | stable sample | R2 | 证明单包也是合法结构 |
| subpacks | Pack 层 | preview 结构 | 为 pack 提供可选子包结构 | 需要 preview schema 的 pack | preview | O5-SCHEMA, O5-SUBPACK | 不应写成默认稳定结构 |

## 8. 结构写作准则

后续继续写结构文档时，任何项都应先判断它属于哪一类：

1. **stable**：官方工作流或官方稳定文档明确支持。
2. **preview**：官方 preview / schema 语义。
3. **sample-only**：真实样例中出现，但不能偷换成官方稳定标准入口。

同时必须避免两种混淆：

1. 把 **manifest 字段** 和 **目录/文件结构** 混成一类。
2. 把 **官方 starter 骨架**、**官方样本库可见结构** 和 **样例特有路径** 混写成同等标准。
