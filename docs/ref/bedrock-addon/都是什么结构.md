# Bedrock Add-On 都是什么结构

## 1. 范围说明

这里的“结构”不是指某一个 JSON 文件内部的字段树，而是指 Bedrock Add-On 从外到内的层级结构：

1. 分发容器长什么样。
2. 一个 addon 里有几个 pack。
3. Resource Pack 和 Behavior Pack 各自有哪些目录族。
4. 真实样例里这些结构通常怎么组合。

## 2. 先看总层级：Bedrock Add-On 至少有四层结构

第一层到第四层可以这样理解：

```text
Add-On
└── 容器层
    ├── 文件夹
    ├── .mcpack
    └── .mcaddon
        └── Pack 层
            ├── Behavior Pack
            ├── Resource Pack
            └── 可能还有 script module / template module
                └── 内容层
                    ├── manifest.json
                    ├── pack_icon.png
                    ├── 目录族（entities, models, textures, ...）
                    └── 各目录内的具体 JSON / 资源文件
```

这也是为什么“只会读几个 `entity` 和 `animation` 文件”远远不能算“理解 addon 结构”。

## 3. 容器层结构

### 3.1 文件夹结构

官方文档和社区文档都确认，开发态最常见的是直接以文件夹形式存在，放在：

- `development_behavior_packs/`
- `development_resource_packs/`

同时，稳定导入后的 pack 会出现在：

- `behavior_packs/`
- `resource_packs/`

这意味着“一个 addon”在开发时，本质上往往就是一对或多对文件夹。

### 3.2 `.mcpack` 结构

`.mcpack` 可以理解为“单个 pack 的压缩容器”，通常等价于把一个 BP 或 RP 目录打包后换扩展名。

它本身并不天然意味着“完整 addon”，更像是“单个资源包或行为包的可导入分发格式”。

### 3.3 `.mcaddon` 结构

`.mcaddon` 是复合容器。社区文档对这一点说得更直白：它本质上是一个复合归档，可以一次导入多个资产归档。

所以从结构上看：

```text
.mcaddon
├── 一个或多个 .mcpack
└── 或者一个或多个顶层 pack 目录
```

这也是为什么“只支持单目录”并不等于支持完整 addon。

## 4. pack 层结构

一个真实 Bedrock addon 往往不是一个包，而是一组互相依赖的 pack。

最常见的组合是：

```text
Addon
├── Behavior Pack
└── Resource Pack
```

更复杂的样例还会包含：

- `script` module
- 额外的脚本依赖
- 更完整的元数据
- 特殊 capability

## 5. Resource Pack 的结构

### 5.1 官方教程和工作流里明确出现的 RP 目录族

官方工作流和资源包介绍明确给出过这些目录族或文件：

- `animations/`
- `animation_controllers/`
- `entity/`
- `models/blocks/`
- `models/entity/`
- `render_controllers/`
- `textures/blocks/`
- `textures/entity/`
- `textures/items/`
- `texts/`
- `manifest.json`
- `pack_icon.png`

这说明 RP 绝不是“只有 textures”。

### 5.2 Mojang 官方样本库里出现的更完整 RP 目录族

`Mojang/bedrock-samples` 的 `resource_pack/` 目录进一步展示了更大的版图，至少出现了：

- `animation_controllers/`
- `animations/`
- `attachables/`
- `biomes/`
- `entity/`
- `fogs/`
- `models/`
- `particles/`
- `render_controllers/`
- `sounds/`
- `texts/`
- `textures/`
- `ui/`
- `biomes_client.json`
- `blocks.json`
- `sounds.json`
- `manifest.json`
- `pack_icon.png`

这说明“真实 RP 结构”比教程页更宽，而且包含不少客户端资源系统。

### 5.3 `textures/` 自己还是一个大子树

官方样本里 `textures/` 下面还继续分成：

- `blocks/`
- `entity/`
- `environment/`
- `gui/`
- `items/`
- `map/`
- `misc/`
- `models/armor/`
- `painting/`
- `particle/`
- `particles/`
- `persona_thumbnails/`
- `trims/`
- `ui/`

同时还存在许多顶层纹理索引文件，例如：

- `item_texture.json`
- `terrain_texture.json`
- `texture_list.json`
- `flipbook_textures.json`

## 6. Behavior Pack 的结构

### 6.1 官方文档里明确出现的 BP 目录族

Behavior Pack 介绍和工作流中明确出现过：

- `blocks/`
- `entities/`
- `items/`
- `recipes/`
- `spawn_rules/`
- `structures/`
- `scripts/`
- `texts/`
- `manifest.json`
- `pack_icon.png`

官方教程还明确把玩法类数据总结为：

- entities
- loot
- spawn rules
- items
- recipes
- trade tables

### 6.2 Mojang 官方样本库里出现的更完整 BP 目录族

`Mojang/bedrock-samples` 的 `behavior_pack/` 目录至少出现了：

- `biomes/`
- `entities/`
- `items/`
- `loot_tables/`
- `recipes/`
- `spawn_rules/`
- `trading/`
- `manifest.json`
- `pack_icon.png`

这说明真实 BP 结构也比“只看 entity”宽得多。

## 7. 真实 addon 的结构差异

### 7.1 最小样例和完整样例不是一个量级

从真实样例看，至少可以分成三层：

| 类型 | 典型代表 | 结构特征 |
|---|---|---|
| 最小教程包 | `behavior_pack_sample`、`resource_pack_sample` | 单包、目录少、只演示核心路径 |
| 标准成对 addon | `addon_starter/complete` | 同时包含 BP 和 RP，带 dependency，常有 script module |
| 较复杂真实 addon | `gray_wave`、`Canopy`、`Construct` | 目录族多，manifest 更丰富，脚本依赖更多，可能声明 capability |

### 7.2 真实样例中常见的结构特征

真实样例里反复出现这些结构模式：

1. **BP 与 RP 成对出现**。
2. **BP 里经常带 script module**。
3. **RP 里不仅有 textures，还会有 model / render_controller / particle / attachable / text / sound 等客户端资源族**。
4. **BP 里不仅有 entities，还会有 items / recipes / loot_tables / feature_rules / features / blocks 等玩法资源族**。

## 8. 官方与社区对结构的补充关系

### 8.1 官方文档更像“规范骨架”

官方文档主要负责回答：

- 这些 pack 是什么。
- manifest 该怎么写。
- Resource Pack / Behavior Pack 的典型目录该怎么起步。

### 8.2 社区文档更像“导入规则和实际工作流解释”

社区文档补充了很多官方文档里说得不够直白的事情，例如：

- `mcpack` / `mcaddon` 其实都是 zip 变体。
- `mcaddon` 是复合归档，而不是单一资源包。
- 开发目录与稳定导入目录的区别。
- manifest v3 在实际创作流程里的使用习惯。

### 8.3 真实样例则告诉我们“现实里大家到底怎么组织”

真实样例的价值在于：

- 可以看到最小结构和完整结构之间的实际跨度。
- 可以看到官方教程没列全的目录族。
- 可以看到 manifest 在现实项目中的组合方式，例如 `script + dependencies + metadata + capabilities`。

## 9. 对当前项目研究的启示

如果把 Bedrock addon 的完整结构当成目标，那么至少不能把项目理解限制在：

- `client_entity`
- `animation`
- `animation_controller`
- `model`
- `render_controller`
- `particle`

这些只覆盖了 **一部分 Resource Pack 结构**，离完整 addon 仍然差很多。完整 addon 研究至少还要继续面对：

- BP 侧的 `items / blocks / recipes / loot_tables / spawn_rules / trading / features / feature_rules / structures`
- RP 侧更广的 `sounds / ui / texts / fogs / biomes / texture 索引文件`
- pack 之间的依赖关系与脚本模块结构

## 10. 第一版结论

Bedrock Add-On 的“结构”可以概括成一句话：

> 它不是单一 JSON 格式，而是“复合容器 + 多 pack + 多目录族 + 多类资源文件”的分层系统。

所以，任何声称“已经支持 addon”的实现，都必须先回答清楚：

1. 支持的是 **哪一层结构**。
2. 支持的是 **哪些 pack 类型**。
3. 支持的是 **哪些目录族**。
4. 是按 **文件级结构** 保留，还是已经压扁成项目内部视图。

本页只完成第一阶段，把“都是什么结构”先立出层级和边界。
